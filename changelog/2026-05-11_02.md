## Why Care?

The splash that landed yesterday looked great locally and built cleanly — `pnpm build` produced a `dist/` ready to ship. The instinct was to enable GitHub Pages and watch it go live. The right move was to slow down for ninety seconds and check what Pages would actually serve, because the repo name and the splash's configured base path didn't match. A blind deploy would have published a site where every internal link, every preloaded stylesheet, every image, every sitemap entry, and every OpenGraph URL pointed at a path that didn't exist.

This entry fixes the mismatch, adds the deploy workflow, and runs a clean build to verify every absolute URL now resolves to the right place. Pages is now safe to enable.

## What's New?

- **`.github/workflows/pages.yml`** — runs on push to `master`, `main`, or `development` (and `workflow_dispatch`). Checks out without submodules (the splash uses pre-synced `src/rollup/`, not live submodule reads), installs with `pnpm install --ignore-workspace`, builds, and deploys via `actions/deploy-pages@v4`. `actions/configure-pages@v5` runs with `enablement: true` so the workflow bootstraps Pages itself on first run — no manual UI step required.
- **Base path corrected from `/ai-labs/` → `/lossless-ai-labs/`** in `splash/astro.config.mjs`, `splash/src/lib/seo.ts` (`SITE_URL` *and* `SITE_REPO`), and `splash/public/robots.txt` (sitemap pointer). All four were silently wrong before tonight.
- **README updated** — removed the "workflow will follow in the next iteration" note; documented the deploy trigger and the two acknowledged gaps versus the `maintain-splash-pages` skill's full convention (Pagefind search and `/llms.txt` endpoints — deferred, not deploy blockers).
- **Clean build verified.** After the path edits, `pnpm build` from a wiped `dist/` produced 40 pages, all OG image URLs absolute and pointing at `https://lossless-group.github.io/lossless-ai-labs/...`, sitemap index and child sitemap both correctly rooted at `lossless-ai-labs`, canonical link correct, robots.txt sitemap pointer correct.

## The bug that was about to ship

Pages serves at `https://lossless-group.github.io/<repo>/`. The repo is `lossless-ai-labs`, so Pages would serve at `https://lossless-group.github.io/lossless-ai-labs/`. But `splash/astro.config.mjs` had:

```js
site: 'https://lossless-group.github.io',
base: '/ai-labs/',
```

…and Astro builds *every* internal `<a>`, `<link>`, `<script>`, `<img>`, sitemap entry, and OG image URL by prepending `BASE_URL` to the path. So the rendered HTML would have shipped with hundreds of `/ai-labs/...` references, every one of them 404 on a host serving at `/lossless-ai-labs/`. The favicon, the CSS bundle, the OG image, the in-page nav — all broken.

The local directory is `ai-labs/` (legacy from before the repo was created with the `lossless-` prefix), so when scaffolding the splash yesterday the obvious-looking value for `base` was the directory name. Worked on `localhost`, would have failed on Pages. The asymmetry was invisible because nothing in the splash test loop touched the deployed URL.

The same mismatch lived in two other places:

```ts
// splash/src/lib/seo.ts — before
export const SITE_URL = 'https://lossless-group.github.io/ai-labs/';
export const SITE_REPO = 'https://github.com/lossless-group/ai-labs';
```

`SITE_URL` feeds canonical links and the OG sextet's absolute URLs. Unfurlers read those values verbatim — a wrong `og:image` URL means iMessage / WhatsApp / Slack all show the favicon fallback (or nothing) instead of the 1,312×736 instrument-bench banner that just got generated. `SITE_REPO` was wrong in a quieter way — the "github" link in the footer would have 404'd.

```
# splash/public/robots.txt — before
Sitemap: https://lossless-group.github.io/ai-labs/sitemap-index.xml
```

Sitemap discovery convention says crawlers look at robots.txt for the `Sitemap:` line. A wrong absolute URL there is the kind of issue you only catch by submitting the sitemap manually to Google Search Console and watching it report 404 — by which point you've already wasted indexing budget.

Fix in all four places is one search-and-replace: `/ai-labs/` → `/lossless-ai-labs/`, `lossless-group/ai-labs` → `lossless-group/lossless-ai-labs`.

## What's not yet wired (intentional follow-ups)

These were the splash's three gaps versus the `maintain-splash-pages` skill's full convention before this entry, and they remain gaps after:

1. **No Pagefind search.** The skill's convention is "every splash ships search-by-default" — confirmed twice across the family (memopop-site, lfm/splash). The ai-labs splash will need `astro-pagefind` + the `data-pagefind-body` markers on detail pages, a `SearchBox` component, a `/search` route, and CSS-variable overrides on the Pagefind UI. This was deferred from yesterday's scaffold; it's still deferred.
2. **No `/llms.txt` + `/llms-full.txt` endpoints.** Context-vigilance-kit (2026-05-09) and memopop-ai (2026-05-11) both now serve these for LLM-facing discovery. The `astro.config.mjs` here already excludes them from the sitemap filter (forward-looking), but the actual endpoints don't exist yet.
3. **No sort controls on `/changelog/` and `/context-v/` list pages.** The skill specifies a `SortControls` component, server-pre-sorted default, per-page `localStorage` persistence. Not present here yet.

All three are content-quality additions, not deploy blockers. Pages goes live cleanly without them; they get added in subsequent passes when the splash is ready for that level of polish.
