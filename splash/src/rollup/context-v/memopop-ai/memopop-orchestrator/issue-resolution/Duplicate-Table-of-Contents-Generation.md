# Duplicate Table of Contents Generation

**Status**: Diagnosed, partially fixed
**Date**: 2026-03-23
**Severity**: Medium — cosmetic but client-facing

## Problem

When regenerating the TOC and re-exporting a memo, multiple duplicate TOC blocks appear in the final HTML/PDF output. In one observed case (Solugen v0.0.1), four separate TOC blocks appeared in the rendered document.

## Root Cause Analysis

There are **three independent mechanisms** that can each insert a TOC, and they are not fully coordinated:

### 1. `insert_toc_after_executive_summary()` inserts without checking for existing TOC first

In `src/agents/toc_generator.py`, the `insert_toc_after_executive_summary()` function is called after `remove_existing_toc()` strips the old one. However, its insertion logic searches for the position *after the first `---` separator* as a fallback (line 168-172). If the document has multiple `---` separators (common in memos with HR rules between sections), or if the Executive Summary heading is not found, the fallback can insert at unexpected positions — and if called multiple times, each call adds another TOC.

### 2. `remove_existing_toc()` regex may not match all TOC variants

The removal regex (`r'## Table of Contents\n(?:[ \t]*(?:-|\d+\.)[^\n]*\n)*\n*(?:---\n*)?'`) assumes the TOC is immediately followed by its list items with no blank lines between the heading and the first item. If there's a blank line after `## Table of Contents` (as generated by `generate_toc_markdown()` which adds `\n` after the heading), the regex won't match and the old TOC survives removal. The new TOC then gets inserted alongside the old one.

**Specific mismatch**: `generate_toc_markdown()` outputs:
```
## Table of Contents\n\n1. [First Item]...
```

But `remove_existing_toc()` expects:
```
## Table of Contents\n1. [First Item]...
```

The `\n` gap between heading and first list item causes the regex to fail silently.

### 3. Multiple regeneration cycles compound the problem

Each call to the TOC generator that fails to fully remove the previous TOC creates an additional copy. Over several export iterations (e.g., fixing citations, renaming scorecard, re-exporting), the duplicates accumulate. The validation check in the agent (lines 272-286) compares TOC content but may not catch positional issues.

## Fixes Applied (2026-03-23)

1. **Updated `remove_existing_toc()` regex** to match both bullet (`-`) and numbered (`1.`) list markers, since the TOC format was changed from bullets to numbered lists.

2. **Updated `extract_existing_toc()` regex** with the same pattern fix.

3. **Changed TOC format** from flat unordered bullet list to numbered ordered list with proper indentation reflecting header hierarchy (h1 → h2 → h3).

4. **Added h1 header extraction** — the previous `extract_headers()` only captured h2 and h3 headers, missing h1 (`#`) headers which are the primary section headers in Alpha Partners 7Cs memos.

5. **Collapsed depth gaps** — if a document uses h1 and h3 but no h2, the TOC now maps to sequential depth levels (0, 1) instead of literal levels (0, 2), preventing excessive indentation.

## Remaining Risks

### The regex-based removal is inherently fragile

The `remove_existing_toc()` function relies on exact formatting expectations. Any variation in whitespace, blank lines, or list formatting between the generator output and the removal regex will cause silent failures. Consider replacing with a stateful parser that finds the `## Table of Contents` line and removes everything until the next `##` heading or `---` separator.

### No idempotency guard on insertion

`insert_toc_after_executive_summary()` does not check whether a TOC already exists in the content before inserting. The current pattern of "remove then insert" works only if removal succeeds completely. A safer approach would be to check for `## Table of Contents` in the content before insertion and abort if found.

### Proposed fix: stateful removal

Replace the regex approach with a line-by-line parser:

```python
def remove_existing_toc(content: str) -> str:
    lines = content.split('\n')
    result = []
    in_toc = False
    for line in lines:
        if re.match(r'^## Table of Contents', line):
            in_toc = True
            continue
        if in_toc:
            # Still in TOC if line is a list item, blank, or ---
            if re.match(r'^(\s*(?:-|\d+\.)\s|---\s*$|\s*$)', line):
                continue
            else:
                in_toc = False
        if not in_toc:
            result.append(line)
    return '\n'.join(result)
```

This approach is immune to whitespace variations and handles any list format.

## How to Verify

After any TOC regeneration, grep for duplicate headings:

```bash
grep -c '## Table of Contents' output/Company-v0.0.x/final-draft.md
# Should return exactly 1
```

If the count is >1, duplicates are present and need manual cleanup before export.
