## Why Care?

The deck the firm walks into an LP meeting with should look like a **printed annual
report or a high-end gallery wall**, not a stack of slide wireframes. Two prior
approaches did not produce that:

- **The variant-cycling 16:9 letterbox** (`2026-04-30_01`) was useful as compositional
  recon, but the constrained canvas plus `vw`-unit sizing fought every typography
  decision, costing more iteration time than it saved.
- **The wireframe-first pattern** in general was burning effort on layout decisions
  that should have been informed by content and design tokens, not invented at the
  same time as them.

This day's work is the **third approach**, and the one that actually produces
LP-shippable quality on the first build. The pattern (borrowed from `dark-matter`'s
`/thesis` page) is:

1. Each slide is a section of a single page (`min-height: 100vh`), not a fixed-aspect
   canvas. Typography breathes at natural viewport scale — no glyph-metric guessing,
   no `vw` units fighting a letterbox.
2. Each section is a one-purpose component that **composes a small vocabulary of
   utility classes** defined once in `theme.css` (`.eyebrow`, `.headline`,
   `.section-title`, `.subtitle`, `.statement`, `.stat-large`, `.badge`, `.card`).
   Sections add their own scoped styles only for unique flourishes.
3. The deck is a single page with scroll-snap and arrow-key navigation. The
   slideshow feel is preserved without the deck-engine fragility.

The route lives at `/thesis` (matching the dark-matter naming convention so the two
are comparable). It plays start-to-finish with arrow keys. It scrolls cleanly. The
typography is at adult sizes. Stats land. Type pulls weight. Flat statement: the
deck looks like a deck.

The reason this matters to Calm/Storm: this is the build the firm can show an LP this
week, with copy edits made directly in the narrative markdown files and visual
adjustments made by re-pointing one or two semantic tokens.

## What Was Built

### `src/styles/theme.css` — the token cascade

A complete light-mode theme implementing the two-tier color system from `DESIGN.md`,
with concrete wirings I committed since the `semantic_colors` block in the spec is
still empty. The file has four layers:

1. **Tier 1 named palette** — the named colors components are not allowed to read.
   Subset of the 44-color palette with the entries actually used by current sections.
2. **Tier 2 semantic roles** — the layer components consume:
   - Surfaces: `--color-surface`, `--color-surface-warm` (cream), `--color-surface-muted`,
     `--color-surface-soft` (pale blue tint)
   - Text: `--color-on-surface`, `--color-on-surface-strong`, `--color-on-surface-muted`,
     `--color-on-surface-dim`, `--color-on-surface-faint` (a 5-step ladder)
   - Outlines: `--color-outline`, `--color-outline-strong`, `--color-outline-faint`
   - Brand: `--color-primary` → `blue_calm-storm` (the firm's signature dark blue),
     `--color-primary-container`, `--color-on-primary`
   - State: `--color-state-error`, `--color-state-warning`, `--color-state-success`,
     `--color-state-highlight`
   - Tones: `--color-tone-warm` (coral), `--color-tone-cool` (teal-deep)
3. **Effect tokens** derived from the palette: `--shadow-sm/md/lg`, `--glow-primary`,
   `--overlay`, all using `rgb(15 18 21 / X)` so they share the `black_bunker` base.
4. **Typography + utility classes** — `.eyebrow`, `.headline` (with `.headline-display`,
   `.headline-tight`, `.headline-medium` modifiers), `.section-title`, `.subtitle`,
   `.statement`, `.caption-cap`, `.tagline`, `.wordmark`, `.stat-large`, `.stat-medium`,
   `.stat-small`, `.stat-label`, `.badge` (with `.badge-warning/error/success/warm/confidential`
   variants), `.card`, `.avatar-initials` (with `--lg` and `--sm` sizes),
   `.affiliation-chip`, `.rule-strong`. Plus the `.slide` structure (`min-height:
   100vh`, scroll-snap-align, padding) and reveal-animation system (`.reveal-item`
   with `data-reveal` and `--delay` CSS variable).

### `src/layouts/PageAsDeckWrapper.astro` — the deck wrapper

Adapted from `sites/dark-matter/src/layouts/PageAsDeckWrapper.astro`, light-mode
themed. Provides:

- Scroll-snap container with mandatory y-axis snapping.
- **Keyboard navigation** — ← / → / Home / End / PageUp / PageDown with
  contenteditable detection so it doesn't fire while typing.
- **Double-click navigation** — upper half goes back, lower half goes forward.
- **Section indicator** — fixed bottom-right `current / total` counter that updates
  on scroll.
- **IntersectionObserver** firing reveal animations on `.reveal-item` children with
  per-item `--delay` support.
- **Hint** — single bottom-center "scroll · arrow keys · double-click to advance"
  pill that fades out after 2.4s.

Light-mode adaptation: the dark-matter version used black-on-white pill-shaped chrome;
this version uses white-on-soft-blue with the `--color-outline` border and Calm/Storm's
muted text colors so the chrome belongs to the brand.

## What Changed in Approach (the meta-lesson)

The reason the AI-assisted output went from struggling-to-converge to first-pass-right
is the **upstream context the model had to work with**:

| Previous mode | This mode |
|---|---|
| Improvise layout in a 16:9 letterboxed canvas | Compose section into a `min-height: 100vh` slot |
| Extract copy from PDF mid-build | Read pre-extracted, pre-sharpened copy from `context-v/narratives/` |
| Make typography decisions per slide | Reach for `.eyebrow`, `.headline`, `.section-title` from a defined vocabulary |
| Reason about color via hex values | Reach for `--color-primary`, `--color-on-surface-strong` from a token cascade |
| Treat each slide as a one-off | Compose against a reference implementation (`/thesis` from dark-matter) |
| Use `vw` units inside a constrained canvas | Use `clamp()` against the actual viewport |
| Iterate on visual until it lands | Inherit visual decisions from theme.css; iterate only on composition |

The lesson is portable beyond this project: **AI-assisted design and authoring works
much better when the model is composing against pre-decided vocabulary and pre-written
content, rather than inventing both alongside the layout.**

## Open Items

- **`semantic_colors` in DESIGN.md is still empty.** Today's `theme.css` implements
  the wiring directly in CSS with opinionated choices. Once the YAML semantic block
  is filled, theme.css can be regenerated from DESIGN.md (or maintained as a
  single source of truth in CSS, with DESIGN.md descriptive only). Decision deferred
  until the semantic mappings are reviewed.
- **Astro 6 Fonts API not yet wired in `astro.config.mjs`.** The site is using a
  system-font fallback for body text and Montserrat-by-stack for display. One config
  edit (per the Typography prose in DESIGN.md) when display + body fonts are confirmed
  loaded.
- **Body family decision pending.** DESIGN.md proposes Inter / Source Sans 3 / Lora as
  candidates. Compare-on-render once we have the Astro Fonts API loading at least one
  body family.
- **No imagery in any slide.** Per Phase 1 spec (no heavy imagery in v0.1.0). Initials
  avatars, text-marks for company names, founder counts in lieu of face grids.
  Slides 3, 12, and 15 are the most affected and will benefit most when imagery is
  added in a later phase.
- **Reveal animations are minimal** — fade + 16px slide-up, respects
  `prefers-reduced-motion`. No layout-shifting transforms.
- **`/thesis` is the canonical play route**, but the deck root at `/` still points at
  the variant-chooser TOC. Consider adding a top-level link from `/` → `/thesis` once
  the deck is signed off.
