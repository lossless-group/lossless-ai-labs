## Why Care?

Private markets run on confidentiality. Fund strategy, portfolio construction,
limited-partner identity, valuation marks, term sheets, deal pipelines, even
the mere fact that a fund is talking to a particular institution — every one
of these is information that belongs to the people who entrusted it. A general
partner does not own LP interest data; they steward it. A venture firm does
not own a founder's confidential metrics; they hold them in trust during the
life of the relationship.

This changelog is about the smallest possible reflection of that principle in
a digital workspace: **no one should arrive at a page of deck material without
first being told they're in the right place.** Even when the URL is brand new,
even when nobody has shared it yet, even when search engines have not yet
indexed it, even when the practical risk of a stray visitor is essentially
zero — the right default is still: a cover, a code, a deliberate act of
entry. Anything less makes the workspace feel public-by-default, which is
exactly the wrong posture for material a fund is preparing to put in front of
sophisticated investors.

The deck content here describes a fund's positioning, team, competitive
landscape, portfolio, returns thesis, and proposed terms. It is not yet
finalised. It is not for discovery. It exists in this workspace so that
collaborators with explicit access can iterate on it, and so that prospective
LPs with a code can review it on their own time. That second audience is
small, named, and hand-selected. Everyone else — the entire indistinguishable
remainder of the internet — should see a cover, nothing more.

This is the v1 of an access posture that the deck workspace will carry
forward. It is not the most elaborate possible posture, and it does not need
to be. What matters at this stage is that **the default is closed**. We can
tighten the gate over time as the audience grows; we cannot retroactively make
a previously-public page private.

## What Was Built

### The cover at `/`

The site root is now a deliberate front door, not a directory listing. It
presents the Calm/Storm wordmark on the firm's signature dark blue, an
eyebrow ("Fund III · Restricted Access"), and a single password field labeled
"Enter access code." The act of entry is intentional and unmistakable:
something is on the other side; entering the code is how you get there.

A correct code unlocks the workspace for the rest of the browsing session.
The unlocked menu — the slide-by-slide variant chooser, the assembled scroll
decks, the changelog — appears in place of the cover, with the full site
chrome restored. An incorrect code clears the input and shows a friendly,
inline error. There is no public path around the cover.

### A gate on every deck page

Every page that contains deck material includes a synchronous gate check that
runs before any content paints. If the unlock has not been completed, the
visitor is redirected back to the cover. This is true regardless of how the
page was reached: typed URL, bookmark, link from email, social share. There
is no surface inside the workspace where deck material renders to a visitor
who has not first entered the code on the cover.

The same mechanism is applied uniformly across:

- the slide-by-slide variant chooser pages (`/{slug}`)
- every individual variant draft (`/drafts/{slug}/{slug}-vN`)
- the assembled full-deck scroll variants (`/thesis`, `/thesis/version-2`)
- this changelog itself (`/changelog`, `/changelog/{slug}`)

The cover page (`/`) is the only entry point. Because the gate is wired into
the shared site header, every existing and future page that uses that header
inherits the protection automatically. New pages do not require new wiring.

### Crawlers explicitly disallowed

We are belt-and-suspenders about machine readers. Three layers tell automated
agents that this site is not for them:

1. A `robots.txt` at the site root declaring `Disallow: /` for every
   user-agent. Compliant crawlers (Google, Bing, Sifted-style aggregators,
   archival bots) will respect this and skip the entire site.
2. A `<meta name="robots" content="noindex, nofollow, noarchive, nosnippet">`
   tag on every rendered HTML page, so a crawler that fetches a page despite
   the disallow still sees an explicit request not to index, follow, archive,
   or snippet its contents.
3. An `X-Robots-Tag: noindex, nofollow, noarchive, nosnippet` HTTP response
   header set globally at the edge via Vercel configuration, so even crawlers
   that don't parse HTML but do honour HTTP headers receive the same
   instruction.

This is overkill for a brand-new URL that nothing links to. It is also the
right baseline. Any of these three layers acting alone would do most of the
work; together they make it unambiguous that automated indexing of any
content under this domain is not consented to.

### Configuration in environment

The unlock code lives in an environment variable (`PUBLIC_DECK_CODE`),
managed via `.env` for local development and via the Vercel project settings
for production. This means the code can be rotated without a code change, and
different environments can use different codes. A `.env.example` file is
committed to document the convention; the real `.env` is git-ignored.

## A Note on Posture

A few sentences on what this gate is *for*, separate from what it is.

The first audience for this workspace is the small group of people inside the
firm and a handful of trusted collaborators who are helping prepare the
materials. The second audience, beginning soon, is a hand-picked set of
prospective LPs being invited to look. Both groups receive the URL and the
code through deliberate, traceable channels. Neither group reaches the
material by stumbling onto it.

Outside those two groups, the gate is what the workspace tells everyone else:
**this is not for you, and we are not pretending otherwise.** That is a
position. It is the same position any well-run fund takes with respect to its
documents: closed by default, opened by invitation, never offered as bait to
the public web. That a single shared code is sufficient for v1 reflects the
small size and trusted nature of the audience today; it does not reflect any
ambivalence about the principle.

The principle is durable. The mechanism will evolve as the audience grows —
per-LP codes, time-bounded sessions, audit trails, and stronger transport
guarantees are all tractable when there is a real need for them. None of
those are needed at the scale of v1. All of them are foreseeable. **The
important thing is that the workspace begins from a closed posture rather
than backing into one later.**

## Open Items (the deliberate next steps)

These are not gaps in v1. They are the path of evolution we already see, and
which we will commit to as the audience grows.

- **Per-LP access tokens.** A single shared code is appropriate when the
  audience is small and the relationships are direct. As the deck reaches
  more institutions, individual codes per LP — with the ability to revoke or
  rotate any single one without disturbing the others — becomes the
  natural next step.
- **Server-verified entry.** The current gate is checked in the browser. As
  soon as we want to be able to deny content to a visitor who already has the
  URL but should no longer be approved, we will move the verification into a
  server middleware. The blueprint for this transition is documented in
  `context-v/explorations/Gate-Sensitive-Information-with-Simple-Code.md`
  (Option C). It is a one-hour change when the moment arrives.
- **Session lifetime controls.** v1 leaves the unlock in place until the
  visitor clears their browser storage. A bounded session (24 hours, 7 days,
  configurable per environment) is straightforward and worth adopting once
  the audience extends beyond direct collaborators.
- **Audit on entry.** Logging successful and failed entry attempts becomes
  meaningful when the audience is large enough that knowing *who* accessed
  the workspace and *when* materially serves the firm's stewardship of the
  material. v1 does not need this; v2 might.
- **Production environment configuration.** The `PUBLIC_DECK_CODE` value
  must be set in Vercel's environment variable panel for the deployed site
  to honour the chosen code. Local development works against `.env`.
