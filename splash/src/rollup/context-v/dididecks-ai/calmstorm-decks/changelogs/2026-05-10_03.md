## Why Care?

A few hours ago we shipped the slide tier — 17 slot families, three variants each (51 candidate slides), all adapted from the corresponding scroll-deck sections. The next problem became immediately obvious: how does a reviewer actually walk through 51 slides and *decide which ones are good enough to use*? Without a tracking layer, the review is just "open the player, scroll through, get to slot 09, lose track of which v1 you liked from slot 03." Decisions evaporate.

![Slide Status Manager in Action](https://ik.imagekit.io/xvpgfijuw/Didi-Decks-Embeds/2026-05-10_Didi-Decks_Play--Audit_9.07.17%E2%80%AFPM.png)

Today's change closes that loop. The four-status enum we'd been writing about in prose — `urgent-redo`, `non-urgent-could-be-better`, `passable`, `perfect` — is now a button bar in the per-section player's top chrome. One click per variant, the decision persists to a JSON registry file, and the audit page at `/data-assets/slides` reflects the latest state immediately. The reviewer can play through the whole deck in one sitting, end with a deterministic record of which slides need follow-up, and hand that record to the next session as a punch list.

This isn't an audit concept anymore. It's an audit *tool*.

## What's New?

### `/api/slide-status` — the dev-time write API

A new Astro on-demand route at `src/pages/api/slide-status.ts`. Two methods:

- `GET /api/slide-status` → returns the full status registry as JSON
- `GET /api/slide-status?slide={key}` → returns one entry's status (or `pending` if unset)
- `POST /api/slide-status` body `{ slide, status }` → upserts an entry, returns the updated registry

The route is marked `export const prerender = false`, so it's an on-demand server endpoint. In `pnpm dev` it works as expected; in the Vercel build it gets bundled as a serverless function automatically (we noticed the build log starting to include `Bundling function` for the first time). The audit workflow is dev-time, so the prod function is a side-effect, not a feature requirement.

### Status buttons in the per-section player

The chrome of `/play/section/{slot}` has a new fieldset in the **top** chrome (originally bottom; moved when we noticed Astro's dev toolbar at the bottom of the viewport was sitting underneath it):

```
[Section 12 · Competitive Advantage]   Variant v3   STATUS [Urgent] [Non-urgent] [Passable] [Perfect] [Clear]   3/3
```

Click any button to set the active variant's status. Each status button has its own active-state color so the currently-set choice is unmissable: red (urgent), amber (non-urgent), blue (passable), green (perfect), faint outline (clear → pending). A small toast flashes briefly at the top after each save (`v3: perfect`).

Keyboard shortcuts make the review fast — no need to mouse:

- `←` / `→` / `Space` / `PageUp` / `PageDown` → navigate between variants
- `1` / `2` / `3` / `4` → set urgent / non-urgent / passable / perfect
- `0` → clear status (reset to pending)
- `F` → fullscreen
- `C` → toggle chrome (true presentation mode)
- `Esc` → exit fullscreen

The variant indicator in the chrome updates as you navigate, and the status button highlight reflects the *active variant's* status — so flipping `←` `→` `1` `→` `4` `→` `3` walks through three variants and rates each.

### Updated `/data-assets/slides` audit table

The slides audit page (which previously only showed `adapted` vs `stub` per cell) now stacks two pills per cell when a status is set:

```
v1                    v2                    v3
[adapted]             [adapted]             [adapted]
[passable]            [perfect]             [urgent]
```

Color palette mirrors the player buttons exactly. Bottom of the page now has a counts breakdown across all 51 variants:

```
[urgent] 4   [non-urgent] 12   [passable] 18   [perfect] 9   [pending] 8
```

So a reviewer can see at-a-glance how the deck is shaping up: how many slides are perfect-ready vs how many still need rework. The pills make scanning down the table for "all the urgents" trivially easy — they jump out in red.

### `data/audits/slides.json` — the registry file

All status decisions persist to `data/audits/slides.json`, a flat key→status map sorted alphabetically for clean version-control diffs:

```json
{
  "01-disclaimer-confidential-v1": "passable",
  "01-disclaimer-confidential-v2": "perfect",
  "12-competitive-advantage-v3": "urgent-redo"
}
```

Hand-editable in any editor. Versioned alongside the slides. When a future session re-opens the audit, the previous reviewer's decisions are immediately visible in the player chrome (button highlights) and audit table (status pills).

## The Story

This was a "build it small, then notice what's missing, then fix it small" loop — three iterations in maybe an hour, but each iteration solved a real friction point.

### Iteration 1 — "where do we put the status?"

The first design question was storage. Three real options surfaced:

1. **Slide file frontmatter** — add `review_status:` to each `.astro` file. Awkward to write back from a browser button (would need to parse + edit Astro frontmatter), and lots of files to touch when bulk-clearing.
2. **YAML markdown registry** — `context-v/audits/slide-status.md` with frontmatter listing all entries. Matches the convention used elsewhere in `context-v/`, but needs a YAML parser as a dep.
3. **JSON registry** — `data/audits/slides.json` with `{ slide-key: status }`. Trivial to read/write with `JSON.parse/stringify`, no extra deps, easy to hand-edit.

We picked option 3 for simplicity. The user can still hand-edit the file (it's small JSON, sorted), and the dev-time API is a thin wrapper around `fs.readFile/writeFile`.

### Iteration 2 — "the buttons are at the bottom but Astro's dev toolbar is too"

First version put the status buttons in the bottom chrome of the per-section player, alongside the existing `[← Audit]` and `[‹ Prev section] [Next section ›]` controls. Functionally clean — until the user opened it and the buttons collided with Astro's dev toolbar (the floating icon strip Astro injects at the bottom of every dev page). The buttons sat literally underneath the toolbar.

Two-line fix: moved the entire status fieldset to the **top** chrome (between the section title and the slide counter), reorganized the bottom chrome to be just navigation links, and moved the saved-confirmation toast from `bottom: 4.5rem` to `top: 4rem`. The dev toolbar now sits cleanly in its own zone and the status UX has its own zone. Pure visual hierarchy fix; no logic changed.

### Iteration 3 — "the audit page should reflect what we did in the player"

Initially the audit table at `/data-assets/slides` only showed structural info (`adapted` vs `stub` per variant). With the status registry now persisted, we extended each table cell to *also* render the persisted status as a second pill underneath. Color palette matches the player buttons so the visual vocabulary is consistent across both surfaces.

Bottom of the page got a count summary so a reviewer can see "we have 9 perfects out of 51" at a glance — useful for tracking progress mid-review.

```mermaid
flowchart LR
    Player[/play/section/{slot}<br/>top-chrome buttons]
    API[POST /api/slide-status<br/>on-demand Astro route]
    Registry[(data/audits/slides.json)]
    Audit[/data-assets/slides<br/>colored pills + counts]
    Reviewer((Reviewer))

    Reviewer -->|click 1/2/3/4 or Urgent/Non-urgent/.../Perfect| Player
    Player -->|fetch POST<br/>{ slide, status }| API
    API -->|fs.writeFile JSON| Registry
    Registry -->|fs.readFile at build| Audit
    Reviewer -->|scan colored pills,<br/>see counts at bottom| Audit

    style Player fill:#1d4ed8,color:white,stroke:#1e3a8a
    style Audit fill:#10b981,color:white,stroke:#047857
    style Registry fill:#fef3c7,stroke:#92400e
```

The result is a closed loop: every reviewer keystroke updates a file in the repo. When the reviewer is done, the json file *is* the punch list. When the next session asks "what should I fix?", the answer is `grep urgent-redo data/audits/slides.json`.

## How It Works (Under the Hood)

### The four-status enum, codified once

```ts
// src/pages/api/slide-status.ts
const VALID_STATUSES = new Set([
  "urgent-redo",
  "non-urgent-could-be-better",
  "passable",
  "perfect",
  "pending",     // POST with this clears the entry
]);
```

`pending` is the default for any slide not in the registry. It's also accepted as a POST value — sending `pending` deletes the entry from the registry rather than storing it. Keeps the file small (only meaningful decisions are persisted).

### Optimistic UI with rollback

The player updates its own button state instantly, then POSTs to the API in the background. If the POST fails (e.g. dev server isn't running, or production where the API isn't available), the UI rolls back and shows an error toast:

```ts
async function setStatus(status: string) {
  const key = slideKey(idx);
  const prev = statusRegistry[key];
  // Optimistic update — buttons reflect new state immediately
  if (status === "pending") delete statusRegistry[key];
  else statusRegistry[key] = status;
  renderStatusButtons();

  try {
    const res = await fetch("/api/slide-status", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ slide: key, status }),
    });
    if (!res.ok) throw new Error(await res.text());
    const json = await res.json();
    statusRegistry = json.registry;   // sync to authoritative server state
    showToast(`${key.split("-").slice(-1)[0]}: ${status === "pending" ? "cleared" : status}`);
  } catch (err) {
    // Roll back — restore prior state, show error
    if (prev === undefined) delete statusRegistry[key];
    else statusRegistry[key] = prev;
    renderStatusButtons();
    showToast("Save failed — see console", true);
    console.error("slide-status POST failed:", err);
  }
}
```

The user gets immediate visual feedback (button color flips), the save happens async (no waiting), and if anything goes wrong the UI honestly reflects the failure.

### Slide identity = filename without extension

The registry key for each slide is the filename minus `.astro`:

```
src/slides/by-title/12-competitive-advantage-v3.astro
                  → "12-competitive-advantage-v3"
```

Each slide article in the player carries this as `data-slide-id`, and the JS reads it on click to know what key to POST. The audit page constructs the same key from `(slot, slug, variant)` to look up statuses for display. Single source of truth on the filename → no separate id mapping needed.

### Sorted JSON for clean diffs

When the API writes the registry, it sorts the keys alphabetically before serializing:

```ts
async function writeRegistry(reg: Registry): Promise<void> {
  const sorted: Registry = {};
  for (const k of Object.keys(reg).sort()) sorted[k] = reg[k];
  await fs.writeFile(REGISTRY_PATH, JSON.stringify(sorted, null, 2) + "\n", "utf-8");
}
```

Two reviewers with different review orders end up with the same file content as long as their decisions are the same. Git diffs only show actual decision changes, not key order shuffles.

## What's Next

- **Status notes** — adjacent to each status, store an optional free-form note (`{ status: "urgent-redo", note: "wordmark too small at canvas scale" }`). Lets the reviewer leave context for the next pass without leaving the player. Implementation: add a small text field that appears when a status is set; POST `{ slide, status, note }`.
- **"Next unreviewed" jump** — a `Tab` keystroke (or `n`) that jumps to the next variant whose status is `pending`. Speeds up review when most slides are already classified and only a few remain.
- **Per-slot status rollup** — show the dominant status per slot in the audit table's leftmost column. If 2/3 variants are `perfect` and 1 is `passable`, the slot reads "ready"; if any variant is `urgent`, the slot reads "needs work."
- **"Promote to canonical" action** — once a variant is marked `perfect`, a one-click action could mark it as the canonical for that slot. The full-deck `/play` would then pick that variant instead of the hardcoded v1. Candidate registry: `data/audits/canonical.json` `{ "01": "v3", "02": "v1", ... }`.
- **Export to changelog/PR description** — a button on the audit page that copies a markdown summary of all `urgent-redo` and `non-urgent-could-be-better` items, formatted as a punch list ready to paste into a PR or follow-up issue.

None of those are urgent. The current loop is functional enough that a single review pass produces a usable punch list.
