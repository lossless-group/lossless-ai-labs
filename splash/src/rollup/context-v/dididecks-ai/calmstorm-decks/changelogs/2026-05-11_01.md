# Auth Stack Live in Production

Yesterday's two passes shipped the v2 confidential-access kernel and the operator surface. Today closed the gap between "the code is on disk" and "Lucanus can actually log in," with a handful of integration fixes that surfaced only once we tried to use it end-to-end.

## The fix that mattered

**Tier-3 sign-in was failing with `config_missing` even though `ADMIN_PASSCODE` was set.** The passcode reader (`src/lib/auth/passcode.ts`) was using `process.env.ADMIN_PASSCODE` exclusively. That works on Vercel (which injects real OS env vars at runtime) but **does not work in Astro dev** — Astro/Vite loads `.env` files into `import.meta.env`, not into Node's `process.env`. The server-side code ran, didn't see the variable, returned the "config_missing" error page.

The fix is dual-source reads:

```ts
function readEnv(name: string): string | undefined {
  const fromMeta = (import.meta.env as Record<string, string | undefined>)[name];
  if (fromMeta) return fromMeta;
  return process.env[name];
}
```

Applied to both `passcode.ts` (admin + viewer passcodes) and `token.ts` (`SESSION_SECRET`). Now identical behavior in `pnpm dev` and Vercel runtime — neither environment is the special case. This pattern is worth promoting to the cross-cutting blueprint as a default for any future Astro v6+ server-side env access.

After this patch deployed: `staycalm` at `/access` → admin role granted → cookie set → deck loaded → activity dashboard accessible at `/admin/activity`.

## The Turso env-var-name odyssey

The Turso DB env vars went through four naming generations in the course of getting this working — driven by the user's preference for project-prefixed names (he juggles ~12 projects):

1. `TURSO_DB_URL` / `TURSO_AUTH_TOKEN` (initial)
2. `DIDIDECKS_TURSO_DB_URL` / `DIDIDECKS_TURSO_AUTH_TOKEN` (Vercel rename pass 1)
3. `DIDIDECKS_ASTRO_DB_URL` / `DIDIDECKS_ASTRO_AUTH_TOKEN` (Vercel rename pass 2)
4. `ASTRO_DB_URL` / `ASTRO_DB_AUTH_TOKEN` (local .env current)

`@astrojs/db` itself reads only `ASTRO_DB_REMOTE_URL` + `ASTRO_DB_APP_TOKEN`. The repo now carries a fallback chain in three places — `astro.config.mjs`, the three CLI scripts (`identity-create`, `mint-link`, `seed-clients`), and the new `scripts/db-push.ts` wrapper — that picks the first defined value across all five name variants and aliases it to the names `@astrojs/db` expects. Net effect: whatever name the user has set in any `.env` or Vercel panel, the CLI and the build both find it.

The `db-push.ts` wrapper exists because `astro db push --remote` runs as a standalone CLI subcommand that loads its env *before* `astro.config.mjs` runs, so config-level aliasing doesn't reach it. The wrapper handles env loading via dotenv, aliases, and shells out to the bare command.

## Component foldering cleanup

`TrackerScript.astro` was incorrectly dropped into `src/components/basics/`. Moved to `src/components/auth/`. `basics/` is reserved for genuinely-basic primitives — the tracker is session-bound instrumentation and belongs alongside other auth surfaces. Filed under "fix it when noticed."

## Astro.config env wiring

The aliasing in `astro.config.mjs` is now bidirectional and defensive:

- When `ASTRO_DATABASE_FILE` is set (local dev/build via npm scripts) → forcibly clear any remote-URL env so Astro DB uses the local file. Defends against stale `.env` values triggering the `Invalid URL` crash during prerender.
- When `ASTRO_DATABASE_FILE` is unset (Vercel `--remote` build, `pnpm db:push:remote`, `pnpm build:remote`) → alias whichever Turso env name is present to `ASTRO_DB_REMOTE_URL` / `ASTRO_DB_APP_TOKEN`.
