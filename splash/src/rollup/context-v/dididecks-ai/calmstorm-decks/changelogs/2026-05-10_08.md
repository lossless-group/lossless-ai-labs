# Confidential Access v2 — Operator Surface

Yesterday's pass landed the auth kernel: server-validated passcode, DB-backed sessions, AuthEvent logging. This pass adds everything an operator needs to *use* it — mint a personal link, watch the client click around, and (when needed) revoke.

## CLIs

Both use `tsx` directly and talk to the libSQL DB via `@libsql/client`. They read `.env` automatically (via `dotenv`) and accept `--remote` to write to Turso instead of the local file DB.

```
pnpm identity:create --slug=mp-client-1 --label="Primary Client" --email=...@calmstorm.vc --notes="WhatsApp delivery"
pnpm mint-link --identity=mp-client-1 --role=viewer --days=30 --max-uses=5
```

`mint-link` prints a redemption URL — paste that into the WhatsApp message. The token is HMAC-SHA256 signed with `SESSION_SECRET`; the `MintedToken` row tracks `uses_remaining` and `revoked_at`. The link mints a 90-day rolling session on first click.

Smoke-tested against the local file DB: identity created, link minted, signature verified, payload decoded cleanly.

## /admin/activity dashboard

Server-rendered at `/admin/activity`. Middleware-gated to admin role only; viewer-role sessions get a 404 (so they don't learn the route exists). Renders, newest first:

- **Identities** — every pre-seeded person plus auto-created `shared:<label>` synthetics from the suffix hack
- **Minted tokens** — uses remaining, expiry, revocation state per token
- **Sessions** — last 20, with tier (`signed_link` / `passcode`), role, shared_label, last seen
- **Auth events** — last 50, every attempt success or fail with outcome + reason + hash prefix
- **Page views** — last 50, path + identity + referrer
- **Actions** — last 50, kind + target + payload_json

Plain HTML, no JS, no styling effort beyond legibility. Refresh to update — this is the operator panel, not a product surface.

## /api/track + TrackerScript.astro

The Action sink. POST `{kind, target?, payload?}` as JSON; the endpoint validates `kind` against an allowlist (`slide-viewed`, `section-expanded`, `external-link-clicked`, `scroll-to-bottom`, `tab-focus`, `tab-blur`, `deck-mounted`), caps `target` and `payload_json` lengths, looks up identity + shared_label from the session row, and writes one `Action` row.

`TrackerScript.astro` is a drop-in `<script is:inline>` component that emits the standard set from any page it's mounted on:

- `deck-mounted` once per pageload
- `tab-focus` / `tab-blur` on visibility change
- `scroll-to-bottom` first time near the bottom
- `external-link-clicked` on any anchor whose origin differs from `location.origin`
- `slide-viewed` for any element with `data-track-slide="<slot>"` once it crosses 50% in-viewport (IntersectionObserver)

Uses `navigator.sendBeacon` when available, falls back to `fetch` with `keepalive`. Drops events silently if the session has been revoked.

Not yet wired into the existing deck layouts — opt-in per page. Add `<TrackerScript />` to whichever layouts you want to instrument.

## /api/access/logout

GET or POST. Revokes the `Session` row (sets `revoked_at`), clears the cookie, writes an `AuthEvent` with `reason: "logout"`, redirects to `/access`.

## Env aliasing for Vercel

You added `DIDIDECKS_TURSO_DB_URL` + `DIDIDECKS_TURSO_AUTH_TOKEN` to Vercel (matching the repurposed Turso DB names). Astro DB's `--remote` flag reads `ASTRO_DB_REMOTE_URL` + `ASTRO_DB_APP_TOKEN`. To avoid duplicating env vars, `astro.config.mjs` aliases the former to the latter at process start:

```js
if (!process.env.ASTRO_DB_REMOTE_URL && process.env.DIDIDECKS_TURSO_DB_URL) {
  process.env.ASTRO_DB_REMOTE_URL = process.env.DIDIDECKS_TURSO_DB_URL;
}
if (!process.env.ASTRO_DB_APP_TOKEN && process.env.DIDIDECKS_TURSO_AUTH_TOKEN) {
  process.env.ASTRO_DB_APP_TOKEN = process.env.DIDIDECKS_TURSO_AUTH_TOKEN;
}
```

`vercel.json` `buildCommand` set to `pnpm astro build --remote` so production builds connect to Turso.

## Required one-time setup for production

Before the first Vercel deploy succeeds, push the schema to Turso:

```
pnpm astro db push --remote
```

That creates the six tables in the Turso instance referenced by `DIDIDECKS_TURSO_DB_URL`. If you'd rather isolate from the original Didi project, provision a fresh Turso DB first and swap the two env values.
