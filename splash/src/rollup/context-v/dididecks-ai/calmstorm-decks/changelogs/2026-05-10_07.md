# Confidential Access v2 — Server-Gated, DB-Backed, Telemetered

The v1 access gate was a localStorage flag set by a client-side passcode check. Its own source code carried the disclaimer: *"It is NOT a security boundary."* In practice that wasn't the worst part — the worst part was that when the primary stakeholder couldn't get in last week, we had **no way to tell why**. No server log of attempts. No record of whether his browser even reached the gate JS. No way to distinguish "typed wrong code" from "localStorage blocked" from "didn't open the URL at all."

This commit replaces that with a proper server-side gate, a DB-backed session model, and an event log for every auth attempt.

## What landed

**Spec:** `context-v/specs/Confidential-Access-v2-Persistent-Sessions-and-Telemetry.md` — site-level implementation contract, downstream of the cross-cutting exploration at `astro-knots/context-v/explorations/Rethinking-Confidential-Access-with-Persistent-Sessions-and-Auth-Telemetry.md`.

**Output mode:** `output: "static"` → `output: "server"`. Auth routes, middleware, and DB-touching API endpoints all need SSR. Deck pages that use `getStaticPaths` retain `prerender = true` so they continue to render at build time.

**Schema (Astro DB, `db/config.ts`):** six tables, libSQL-backed locally and Turso-bound in production via the same client API:

- `Identity` — pre-seeded before mint; the "person" that activity attaches to
- `MintedToken` — signed-link issuance ledger, references Identity
- `Session` — live session with role (`admin`/`viewer`), tier (`signed_link`/`passcode`), and `shared_label`
- `AuthEvent` — every verify or redeem attempt, success or failure
- `PageView` — middleware-driven, every gated request
- `Action` — schema present for next-pass interaction tracking

**Auth library (`src/lib/auth/`):**
- `passcode.ts` — parses `<code>` or `<code>-<label>` (suffix hack), matches against `ADMIN_PASSCODE` or `VIEWER_PASSCODE`, returns role + label
- `token.ts` — HMAC-SHA256 signed link tokens; verify checks signature, expiry, and structure; timing-safe comparison
- `session.ts` — 90-day rolling httpOnly cookie helpers; hashing utility for UA/IP
- `types.ts` — shared role/tier/outcome unions

**Middleware (`src/middleware.ts`):**
- Allows `/`, `/changelog`, `/access`, `/api/access/*`, and static assets through
- For everything else: reads session cookie, validates against `Session` row, redirects to `/access?redirect=...` if missing/expired
- Bumps `Session.last_seen_at`, re-issues the cookie (rolling), inserts a `PageView` row
- Returns 404 for `/admin/*` requests from viewer-role sessions (so viewers don't learn admin routes exist)

**Routes:**
- `/access` (SSR) — server-rendered gate page with friendly error rendering for every known failure mode. No JS required.
- `/api/access/verify` (POST) — validates passcode (with suffix), creates synthetic `Identity` for `shared_label`, inserts `Session` + `AuthEvent`, sets cookie, redirects
- `/access/link/[token]` (SSR) — redeems signed link tokens: signature check → DB lookup → revoked/exhausted/expired guards → session mint

**Suffix hack live:** if `VIEWER_PASSCODE=calmbalm`, sharing `calmbalm-anthos` admits as viewer and stores `anthos` on `Session.shared_label`. A synthetic `Identity` (`shared:anthos`) is upserted so all subsequent PageView and Action rows attribute to a label. The dashboard surface that displays this lands next pass.

**Env vars:** `.env.example` updated with `ADMIN_PASSCODE`, `VIEWER_PASSCODE`, `SESSION_SECRET`, and commented placeholders for `ASTRO_DB_REMOTE_URL` / `ASTRO_DB_APP_TOKEN`. The old `PUBLIC_DECK_CODE` stays for now while the v1 client gate coexists.

## Build verification

```
ASTRO_DATABASE_FILE=./astro.db pnpm build
# Server built in ~4.7s, Vercel adapter bundled, no errors.
```

Pre-existing warnings about `Astro.request.headers` on prerendered pages are non-fatal and unrelated to this change.

## Known wrinkle: prerender-vs-middleware

`/play/*` and `/changelog/[slug]` use `getStaticPaths`, which requires `prerender = true`. On Vercel, prerendered pages are served from the CDN and **middleware does not run** for them. So those routes are currently *not* server-gated — they retain only the v1 client gate. Not a regression (same exposure as before), but flagged in the spec as cleanup for the pass that removes the v1 gate. Resolution: refactor the routes to load slide/changelog data at request time inside SSR pages.
