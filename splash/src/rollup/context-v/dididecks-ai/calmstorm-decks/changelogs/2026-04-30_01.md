## Why Care?

A fundraise deck is no good until it is **playable** — until you can sit a partner or an
LP in front of it and click through. Day one of this build delivered exactly that, and
nothing more pretentious. The site at `localhost:4321/` now hosts a real deck with a
deck root, per-slide chooser pages, four to eight compositional variants for the first
four slides, and the navigation chrome (header + bottom-right prev/next + keyboard
shortcuts) you would expect from any slideshow tool.

The reason this matters to the firm is not the variants themselves — most of them will
be shelved. The reason is **iteration speed**. Having 4–8 takes on each opening slide
side-by-side, all reachable by an arrow key, is how a partner can quickly say "that
one, push on it" and the team can converge on a direction in minutes instead of weeks.
The variants are scaffolding for the conversation, not the deliverable.

## What Was Built

### Site chrome

- **`SiteHeader.astro`** — three-column thin header used across every route: wordmark
  on the left (links to `/`), centered "Table of Contents" link, right-side context
  meta ("Fund III · Teaser" or "Slide N · Variant Chooser"). Lives in
  `src/components/SiteHeader.astro` and is imported by every page.
- **`SlideLayout.astro`** — the chrome that wraps every individual slide variant.
  Renders the thin header, a 16:9 letterboxed canvas in the remaining viewport, and
  a low-opacity prev/next nav cluster pinned to the bottom-right with an
  `n / 17 · vK of N` counter that brightens on hover.
- **`global.css`** — minimal foundation; per-slide styling is inline Tailwind on each
  variant (Phase 1 spec: built-in tokens only, no extracted component classes).

### Navigation

- **Bottom-right `‹` and `›` buttons** at low opacity by default, full opacity on
  hover or focus.
- **Keyboard:** ← / → arrow keys click the rendered prev/next anchor (so the same
  anchor logic handles cursor and keyboard).
- **Two navigation modes**, auto-detected from URL:
  - On a chooser page (e.g. `/disclaimer-confidential`): prev/next walks **across
    slides** (slide 1 → slide 2 → …) non-cycling.
  - On a draft page (`/drafts/{slug}/{slug}-vK`): prev/next **cycles through the
    variants** of that slide (v1 → v2 → … → vN → v1). This is the A/B compare flow
    a designer or partner needs when reviewing options.
- **Variant discovery** uses Node `fs.readdirSync` at render time rather than Vite
  `import.meta.glob`. The `import.meta.glob` approach was unreliable in dev mode —
  HMR did not re-evaluate the glob when new variant files were added, so the dev
  server's variant list went stale. `fs.readdirSync` runs every render, no cache.

### Slide content built

Per the spec, every numbered slide gets a slug; every slug gets a variant chooser at
`/{slug}/index.astro` and one or more drafts at `/drafts/{slug}/{slug}-vK.astro`.

| Slide | Slug                        | Variants | Variant names |
|------:|-----------------------------|---------:|---------------|
| 1     | `disclaimer-confidential`   | 4        | Wave · Editorial · Document Head · Field Journal |
| 2     | `vision-mission`            | 4        | Manifesto · Triptych · Compass · Two-part |
| 3     | `venture-team`              | 8        | Card Grid · Roster Table · Stacked Rows · Two by Two · Gallery Wall · Pure Typography · Dossier/Mono · Compass Quadrants |
| 4     | `identity-pillars`          | 5        | Five Pillars · Stats Lead · Editorial List · Wheel · Editorial List · Z-stack |

Source content for slides 3 and 4 was lifted into shared TypeScript modules
(`src/data/venture-team.ts`, `src/data/identity-pillars.ts`) so a single edit
propagates to all variants of that slide. Layouts vary per variant; copy stays in
one place.

### Deck root

- **`/`** — table of contents listing all 17 slides from `src/lib/slides.ts` with
  slug references and a brief instructional caption.

## Outcome and What This Surfaced

The scaffold works. The cycling navigation works. The chooser pages work. The keyboard
bindings work. Build is clean (every variant + chooser + the deck root compiles to
static HTML).

What this round **also surfaced** — and what informed the next day's pivot — is that a
strict 16:9 letterbox inside a desktop viewport ends up squashing typography and
spacing in ways that compound across iterations. Sizing decisions that should have
taken one round took five, because every value was at the mercy of the canvas's
floating dimensions (especially when using `vw` units, which key off the viewport
width but render inside the letterboxed canvas). The variants we shipped are useful
as element-mining material; the **competition between layout choice and canvas
constraint was the real time tax**.

The architecture itself — variant cycling, navigation chrome, content-as-data — is
sound and will carry forward. The 16:9 letterbox constraint is the one we relaxed in
the next build (see `2026-05-01_01`).
