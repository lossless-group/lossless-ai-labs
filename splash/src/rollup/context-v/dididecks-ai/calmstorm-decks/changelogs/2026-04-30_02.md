## Why Care?

The previous day shipped a working scaffold but exposed the cost of building layouts
without a design language. Every iteration was rewriting the same decisions:
*what is primary?* *how dark is the body text?* *should the eyebrow be 0.65rem or
0.75rem?* Quality drifted slide-to-slide. This day's work was the foundation that
makes the next day's deck feel **coherent rather than improvised**.

Three things now exist that did not exist before:

1. **A real design system spec** (`DESIGN.md`) — Calm/Storm's visual identity
   captured in machine-readable tokens plus human-readable rationale. A coding agent
   or a new collaborator can read this file and produce on-brand work without
   guessing.
2. **A complete brand-anchored color palette** (44 named colors, organized by
   recognizable family). The palette is sourced from both the firm's design tool
   export and the live calmstorm.vc site CSS — so what we ship is grounded in the
   firm's actual brand, not invented in our heads.
3. **17 slide narratives** in `context-v/narratives/`. Each is a markdown file with
   structured YAML data (eyebrow, headline, body, key stats, source citations) and
   a markdown interpretation of the slide — what it is, why it's there, what's
   most important to surface, what visual hierarchy serves the LP's read. **The
   copy is now sharpened upstream of the design**, so the design effort can
   concentrate on composition rather than copy-extraction.

The reason this matters to the firm: every minute of iteration time saved here
compounds across 17 slides times some number of clients times some number of decks.
Foundation work is unsexy and necessary.

## What Was Built

### `DESIGN.md` — the design system spec

Adapted from Google's `design.md` template
(`https://github.com/google-labs-code/design.md`) with Calm/Storm-specific extensions:

- **Two-tier color system** instead of the spec's flat `colors:` map. Tier 1
  `named_colers` is the raw hex palette; Tier 2 `semantic_colors` is the role layer
  (`primary`, `surface`, `on-surface`, `error`, …). Components only ever consume
  Tier 2. Re-pointing one Tier 2 token re-themes the system.
- **Obsidian-friendly array syntax** for token blocks (`  - key: value` instead of
  the spec's flat YAML maps). Editable as Properties in Obsidian; trades CLI
  lint-compliance for editorial ergonomics.
- **Effect tokens** (`--shadow-sm/md/lg`, `--glow-primary/accent`, `--overlay`)
  co-located with semantic_colors so the surface stays flat for tooling.
- **Body prose written for three sections** (the substantive ones):
  - **Overview** — what the system serves, the brand premise, audience tone
    (institutional LPs first, founders second), what we explicitly avoid (heavy
    imagery, AI gradients, pure black for text).
  - **Colors** — the two-tier convention explained for both designers and AI
    assistants, family\_funky naming convention, why some hues live outside their
    technical family (cyan_sky, purple_lavender), iteration motion (high-cascade
    first), when to add a new named color.
  - **Typography** — Montserrat as the display family for all five titular tokens
    (display, headline, title, subtitle, eyebrow), body family deferred until
    rendered comparison. Loading guidance updated for Astro 6's built-in Fonts API
    (`fontProviders.fontsource()`, `<Font cssVariable="--font-display" preload />`,
    role-named CSS variables instead of family-named).
- **Empty section headers** scaffolded for Layout, Elevation & Depth, Shapes,
  Components, and Do's and Don'ts — to be filled as decisions are made.

### Color palette — 44 named colors, sourced from the brand

Imports happened in two passes:

**Pass 1 — design-tool export.** The 24-color export from
`draft-palette__CalmStorm.json` was deduped and renamed into `family_funky` form
(`blue_shakespeare`, `grey_bunting`, `cream_horizon`, `coral_dawn`).

**Pass 2 — live site CSS lifted from calmstorm.vc.** 19 additional colors added
from the site's CSS variables (`--midnight-blue`, `--p-classic-blue`, `--orchid`,
etc.). Alpha variants and exact duplicates were filtered out; only the
solid base values made it in. New family prefixes introduced (`cyan_`, `purple_`,
`pink_`, `orange_`) so the blue family stayed scannable instead of ballooning to
17 entries.

The resulting palette is organized **by recognizable color family in
light-to-dark order**:

- 2 whites/cream
- 4 reds/coral/oranges (incl. `coral_dawn`, `red_brick`, `orange_pumpkin`,
  `amber_sunset`)
- 2 yellows (`yellow_paris-daisy`, `yellow_goldenrod`)
- 3 greens (`green_light`, `green_spring`, `green_sage`)
- 3 cyans/teal (`cyan_turquoise`, `cyan_sky`, `teal_deep`)
- 14 blues, anchored by `blue_calm-storm` (#23384d) — the firm's signature dark
  blue, lifted directly from the calmstorm.vc CSS variable `--calm--storm-blue`
- 3 purples (`purple_lavender`, `purple_indigo`, `purple_electric`)
- 1 pink (`pink_orchid`)
- 10 greys (light to dark, weather-progression names: `grey_drizzle` →
  `grey_smoke` → `grey_storm` → `grey_bunting`)
- 2 blacks (`black_bunker` for typography, `black_pure` for the rare full-black case)

### Semantic_colors block — scaffold only

A 52-entry semantic_colors block was scaffolded with empty values, ready for
manual mapping. Roles loosely follow Material 3's palette structure: `surface`
plus surface-container variants, `on-surface` plus dim/strong, `outline`,
`primary`/`secondary`/`tertiary` with on-, container-, and fixed- variants,
state colors (`error`), background, surface-tint. Plus six effect tokens
(`--shadow-sm/md/lg`, `--glow-primary/accent`, `--overlay`).

The wiring was left for human review. The next day's `theme.css` (see
`2026-05-01_01`) implements the wiring directly in CSS using opinionated
defaults, ready to be regenerated from DESIGN.md once the empty values are
filled.

### Typography block — Montserrat for display

A 9-entry typography block was added: `display`, `headline`, `title`, `subtitle`,
`eyebrow` (Montserrat) and `body`, `body-emphasis`, `caption`, `mono` (body
family TBD). Each token has the full Astro-6-Font-API-friendly shape:
`fontFamily`, `fontSize`, `fontWeight`, `lineHeight`, `letterSpacing`. Empty
fields are placeholders pending render-time eyeball decisions.

### Narratives — `context-v/narratives/01-disclaimer-confidential.md` … `17-fund-terms.md`

For every one of the 17 teaser-deck slides, a markdown file was authored with:

**YAML frontmatter — structured content extracted from the source PDF:**
- Slide number and slug
- Section grouping (Cover / Story / Team / Market / Edge / Receipts / Fund)
- `eyebrow`, `headline`, `subhead`
- `key_stat` and `key_stat_label` where applicable
- Per-slide structured fields: `pillars` for slide 4, `columns` for slides 5–6,
  `people` for slides 3 / 11, `terms` for slide 17, `categories` for slide 15,
  `stats_grid` for slide 14, etc.
- `cta_or_footer`, `sources`

**Markdown interpretation — for designers, agents, and reviewers:**
- *What this slide is* — literal description.
- *Why it's here* — the strategic argument; what LP question it answers.
- *What's most important to surface* — visual hierarchy guidance, including which
  lines deserve the most weight (e.g. *"absolute rock, supporting us through
  thick and thin"* on slide 13, *"no IPOs needed for fund model"* on slide 16).
- *Composition risk* — flagged where the slide is hardest to lay out without
  imagery (slides 3, 12, 15 in particular — team, SP grid, portfolio logos).

The narratives ship the deck's voice. They are the copy of record.

## Outcome and What This Enables

DESIGN.md and the narratives became the **spec** that the next day's deck-as-page
implementation read from. The quality difference between the variant-iteration
mode of `2026-04-30_01` and the spec-driven mode of `2026-05-01_01` is directly
attributable to the work done this day.

The DESIGN.md is incomplete (semantic_colors empty, Layout/Shapes/Components/Do's
sections empty), but its **vocabulary is established**. A coding agent can now
read the Overview and Colors sections and know:

- Light-mode only.
- Editorial restraint, no AI-gradients, no heavy imagery.
- Use semantic tokens, not raw palette entries.
- Default to column composition, not glyph-overlap tricks.
- Family scannability matters; use family_funky naming.

The narratives can be edited in Obsidian to sharpen copy without touching CSS.
Edit-then-rebuild is a single round.
