## Why Care?

A fundraise process can't wait on a busy LP rediscovering an access code on
their own time. The polite gate that protects the deck from search crawlers
and casual lurkers is — for an inattentive recipient — exactly the kind of
small friction that means they never see the work at all.

So we built the escape valve: a script that produces high-fidelity PDFs
of the deck, identical to what the LP would see in their own browser if they
*had* unlocked it. They can read the work async, on a plane, on their phone,
without a single login. And because every fundraise / teaser deck workspace
will hit this same friction, the tool is structured to lift cleanly into the
next deck site that follows the calmstorm-decks pattern.

The harder lesson, and the one worth logging: **there is no library that
"gets HTML-to-PDF perfect" for design-heavy decks.** We surveyed eleven
approaches before writing a line of code, and the only one with full
fidelity to what the LP sees in their own Chrome is the same browser the
LP uses — driven headlessly. Anything that re-implements layout (jsPDF,
React-PDF, wkhtmltopdf, html2canvas) drops half the polish. The honest
pipeline is *render in a real browser, walk the DOM, capture, assemble*.
That insight is what `scripts/export-decks.ts` operationalizes.

## What's New?

- **`scripts/export-decks.ts`** — a Playwright-driven deck exporter. One
  pnpm script (`pnpm export:decks`) produces three multi-page PDFs (one
  per scroll deck variant), per-section PNGs underneath, and a 17-page
  PDF of the variant-chooser surface. Lands in `./exports/<ISO-timestamp>/`.
- **`scripts/README.md`** — usage, flags, the silent-killer table
  (font readiness, lazy images, gate bypass, DPR, scroll-snap, reveal
  animations), and a "replicate this in another deck site" section so
  the next site copies cleanly.
- **`context-v/explorations/High-Resolution-High-Fidelity-Deck-Exports-from-Code-to-Images-&-PDFs.md`** —
  the option-space survey (eleven approaches, ordered by ceremony), the
  architectural reasoning, the v1 shipped pipeline, and an honest
  inventory of three known geometric glitches with three forward paths.
- **`package.json`** picked up `playwright`, `pdf-lib`, `tsx`, and
  `@types/node` as devDependencies, and an `export:decks` script.
- **`.gitignore`** now excludes `exports/`. Local artifacts only,
  never committed.

## The arc — three failure modes that turned into three features

This shipped clean on the fourth run. The first three runs each surfaced
a different blind spot, and each fix made the script more robust for the
next deck site to inherit it.

### Run 1 → "the script captured a 404 page from the wrong site"

First export went out, file sizes looked suspiciously small. Opening one
revealed Astro's default 404 page with `Path: /thesis/version-2`. The
server was running, but it was the **wrong server**. With `mpstaton-site`
already bound to `:4321`, calmstorm-decks would auto-pick the next free
port (`:4322`, `:4324`, etc.), and the script's hardcoded
`http://localhost:4321` defaulted to whatever happened to be there.

**The fix:** auto-discovery. Scan ports `4321–4340` in parallel, fetch
each one's homepage, match the body against a `Calm/Storm` fingerprint,
and use the first port that identifies as calmstorm-decks. If nothing
matches, list every port that *did* respond with the title it served, so
the diagnostic actually helps:

```
✗  Could not find calmstorm-decks on any port in 4321-4340.

   Servers responded but none identified as calmstorm-decks:
     :4321  →  Michael P. Staton — VC, Investor, Builder
     :4323  →  404: Not Found

   Start calmstorm-decks:
     cd sites/calmstorm-decks && pnpm preview   (or pnpm dev)
```

This matters beyond calmstorm: running multiple Astro Knots sites in
parallel routinely lights up four to a dozen ports simultaneously, and
the next deck site will hit this immediately. Fingerprint-based discovery
solves it once.

### Run 2 → "the scroll-deck PDFs only contain the first slide"

Captures arrived. Scroll-deck PDFs were proper-looking single-page
documents — except every page only showed slide 1. The other 16
sections were silently missing.

The root cause: the deck wrapper uses `scroll-snap-type: y mandatory`
with each section sized to roughly `100vh`. Crucially, it has its
**own scroll container** — the `.deck-wrapper` div, not the document.
Playwright's `page.screenshot({ fullPage: true })` reads
`document.documentElement.scrollHeight`, which only reflects the
document's outer scroll height. For a scroll-snap deck, that value is
one viewport — the wrapper's internal scroll is invisible to it.

**The fix:** stop trying to capture the whole assembled deck as a tall
document. Walk the DOM instead. Query `section.slide`, scroll each one
into view, capture each as its own PNG, then stitch the per-section PNGs
into a multi-page PDF (one slide per page) via `pdf-lib`. This is
**component-aware export** — it captures the actual rendered section
components, not a serialized approximation.

Two bonus issues surfaced and got fixed in the same pass:

- `scroll-snap-type: y mandatory` *fights* `scrollIntoViewIfNeeded`
  during capture, snapping the wrapper back to the previous section
  mid-screenshot. Solution: `scroll-snap-type: none` injected via
  `addStyleTag` before capture.
- The `[data-reveal]` elements animate in on intersection. Without
  intervention, every section was being captured mid-fade. Solution:
  force-apply `reveal-shown`, `is-visible`, `revealed` classes plus
  `opacity: 1; transform: none` to all `[data-reveal]` nodes before the
  walk begins.

### Run 3 → "the chrome makes the export look unpolished"

Slides came out clean and complete, but the captures included the
DeckHeader at the top, the fixed-position prev/next DeckNav at the
bottom-right, the scroll cue, the slide counter overlay, and (on
chooser pages) the inline prev/next nav at the bottom of `<main>`. All
of which makes the deck a great *interactive* artifact and a poor
*static* one.

The temptation was to thread an `?export=1` query param through the
site and have every page conditionally render chrome. We chose the
opposite: keep export concerns out of runtime code entirely. One
`addStyleTag` call before capture hides everything:

```css
body > header,                 /* DeckHeader */
body > nav,                    /* DeckNav (top-level) */
nav.fixed,                     /* DeckNav (any positioning) */
.nav-hint,                     /* scroll cue */
.section-indicator,            /* slide counter overlay */
main > nav { display: none !important; }
.deck-wrapper { padding-top: 0 !important; }
```

The live site keeps shipping the chrome unchanged. Only the headless
capture session sees this stylesheet. Next deck site that follows the
pattern inherits this behavior for free.

### Run 4 → "good enough for now, not yet 'show off'"

Run 4 produced legible, chrome-free, multi-page PDFs of all three
scroll decks plus the chooser surface. Acceptable for sharing with an
LP who needs the content. Not yet acceptable as a polished
client-facing artifact — and the gap is geometric, not pixel-fidelity.

This is the exact moment to *stop coding and document*, before the
limitations become tribal knowledge. Three glitches, captured frankly
in the exploration doc:

| Glitch | What it looks like | Why it's hard |
|---|---|---|
| Sections aren't 16:9-normalized | PDF pages within one deck have slightly different dimensions; doesn't drop cleanly into PowerPoint or Keynote | Sections were authored for scroll-snap (`--deck-height: calc(100vh - 3rem)`), not for a fixed slide canvas. Forcing 1920×1080 would silently clip any section whose content was scroll-tolerant by design |
| Bleed from neighbors | Thin slice of the previous or next section visible at the top/bottom of each capture | Sections don't enforce `overflow: hidden`. Decorative elements with negative margins or absolute positioning escape the bounding box. The wrapper's gap between sections isn't perfectly zero |
| Sub-pixel scroll alignment | Occasional one-pixel band of the previous section at the top of a capture | `scrollIntoViewIfNeeded` is a heuristic, not a guarantee of pixel-perfect top alignment |

The exploration doc lays out three forward paths — script-injected
export-mode CSS, an opt-in `?export=1` runtime mode, or baking a
strict 16:9 canvas into the design system from day one. Recommended
direction for the next deck site: **Path 3** — make scroll-mode and
export-mode the same artifact, viewed differently. The 16:9 canvas
becomes a section primitive; scroll mode shows the canvas as-is with
the sticky chrome; export mode just hides the chrome. No drift
possible.

For *this* deck workspace, the working plan is asymmetric: PDF exports
for asynchronous LP sharing, plus a **video walkthrough of the live
UI** for the moments where the deck has to actually shine.

## What it enables next

- **The next deck site replicates this in five minutes.** Copy
  `scripts/export-decks.ts`, install three devDeps, point it at a deck
  whose registries match `src/lib/{scroll-decks,slides,gate}.ts`. The
  fingerprint check in `discoverBaseUrl` is the only line that needs to
  change per site.
- **The "Build a Fundraise Deck Workspace" blueprint gets a new chapter.**
  The exploration doc is the source material for adding "deck export
  pipeline" to the playbook for the next workspace, alongside the gate,
  the registry-driven nav, and the OpenGraph system.
- **A future `?export=1` site mode becomes plausible** without backing
  out anything we shipped. The script's chrome-hiding CSS is a thin
  veneer; if we want export mode to be debuggable in the live browser
  later, swap one mechanism for the other and keep going.
