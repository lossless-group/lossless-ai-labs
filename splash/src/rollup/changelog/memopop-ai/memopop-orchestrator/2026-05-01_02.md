# Changelog - 2026-05-01_02

## Milestones Out of the Logs, Files Into the Firm

Yesterday the orchestrator got an HTTP front door. Today it learns to narrate. The FastAPI sidecar now emits structured **milestone** events alongside its raw log stream — clients no longer have to grep for `✅ SECTION RESEARCH COMPLETE` to know research is done; they just listen for `{"type": "milestone", "stage": "research", "level": "success", ...}`. And a missing scaffold step that was sending firm-scoped runs to legacy `output/` is fixed: a firm-scoped `POST /memos` now writes the deal config the orchestrator's path resolver requires, so artifacts land where the user said they should.

Zero changes to the workflow or any agent. `generate_memo()` is identical. `python -m src.main` still works exactly as it did. The CLI doesn't even know any of this happened.

---

## Why

Two distinct problems surfaced when the desktop client (memopop-native) drove its first end-to-end memo through the sidecar.

**The first was the user not being able to tell what was happening.** A live tail of stdout is sexy — and it stays sexy — but it's not what most operators want as their primary signal. Watching the agents write a 67k-character draft over 17 minutes, what they want to know is "where are we, what's left." The orchestrator's agent prints already encode this perfectly (`🔍 PERPLEXITY SECTION RESEARCH (PARALLEL)`, `[3/10] Market Context`, `✅ All sections passed fact-check`) — they just needed to be lifted into structured events with stage and level, so a UI can render them as a checklist instead of asking the user to read the logs.

**The second was a silent fallback bug.** A run was kicked off with `firm="alpha-partners"` and `company_name="ChromaDB"` from the desktop app. Output landed in `output/ChromaDB-v0.0.1/` rather than `io/alpha-partners/deals/ChromaDB/outputs/...`. The cause: `paths.resolve_deal_context()` only honors a passed `firm` if `ctx.exists()` is true — meaning `io/{firm}/deals/{deal}/inputs/deal.json` already exists. The CLI flow assumes you pre-create that file (or `data/{Company}.json` in legacy mode); if it's missing, the resolver silently falls through to legacy output. The CLI tells you in a yellow warning. The API just… did the run, in the wrong place.

The CLI is one user (Mike) who knows the convention. The API surface has to be friendlier than that.

## What Landed

### `src/server/milestones.py` — turning prints into events

A `MilestoneExtractor` with a list of ~30 compiled regex patterns covering the orchestrator's emoji-tagged status prints across ten pipeline stages:

| Stage          | Example agent prints that fire it                           |
| -------------- | ------------------------------------------------------------ |
| `start`        | `📁 Created new output directory`, `📌 Using forced version` |
| `deck_analysis`| `✓ Extracted 23 screenshots`, `✓ Merged analysis complete`   |
| `research`     | `🔍 PERPLEXITY SECTION RESEARCH`, `✓ [03] Market Context: 18 citations`, `✅ SECTION RESEARCH COMPLETE` |
| `competitive`  | `🔍 Researching competitive landscape`, `✓ Evaluation complete: 5 direct, 3 indirect` |
| `writing`      | `📝 Writing memo sections`, `[4/10] Team`, `✅ All 10 sections complete` |
| `enrichment`   | `🔗 Enriching links`, `✓ Team enriched with 8 LinkedIn profiles`, `📊 Generating tables` |
| `assembly`     | `✓ TOC created`, `✓ Reassembled final draft: 12345 words`    |
| `validation`   | `🔍 FACT CHECKING`, `✅ All sections passed fact-check`, `📊 Scorecard Evaluator` |
| `artifacts`    | `✓ Content slots saved`                                      |
| `complete`     | (synthesized client-side from the lifecycle `complete` event)|

Each match returns a JSON-shaped milestone:

```json
{
  "type": "milestone",
  "id": "a3f29c8e1b",
  "stage": "research",
  "level": "info",
  "label": "Researched: Market Context",
  "detail": "18 citations",
  "ts": "2026-05-01T18:42:11.293Z"
}
```

`label` and `detail` are formatted from regex capture groups (`{0}`, `{1}`, ...), so `✓ [03] Market Context: 18 citations` becomes `label="Researched: Market Context"`, `detail="18 citations"`. Static milestones (no captures) are deduped — the orchestrator sometimes prints `✅ SECTION RESEARCH COMPLETE` once at the end of a stage and again in a summary block; the bus only fires it once.

### `src/server/events.py` — `LogSink` now feeds the extractor

`LogSink` accepts an optional `milestone_extractor`. When it splits a log line on `\n` and publishes it as a `log` event, it now also calls `extractor.process(line)`; if a milestone comes back, it's published as a separate `milestone` event on the same `JobEventBus`. SSE subscribers see both event types interleaved in chronological order. The bus's 2000-event backlog covers both.

```python
sink_out = LogSink(bus, mirror=sys.__stdout__, milestone_extractor=extractor)
sink_err = LogSink(bus, mirror=sys.__stderr__, milestone_extractor=extractor)
```

One extractor per job, shared between stdout and stderr — its dedup state is per-run.

### `src/server/app.py::scaffold_firm_deal_dir` — the missing scaffold

`POST /memos` now runs a small idempotent scaffold step before submitting the job:

```python
@app.post("/memos", ...)
async def create_memo(request: CreateMemoRequest) -> CreateMemoResponse:
    scaffold_firm_deal_dir(request)
    job = _registry(app).submit(request)
    return CreateMemoResponse(...)
```

If the request includes a `firm`, it creates `io/{firm}/deals/{company_name}/inputs/` and writes a `deal.json` populated only from fields the request actually carried — `type`, `mode`, `url`, `description`, `stage`, `notes`, `deck`, `dataroom`, `outline`, `scorecard`, `trademark_light`, `trademark_dark`. If `inputs/deal.json` already exists (the user curated it via the CLI, hand-edited, etc.), it's left untouched; only the directory tree is created lazily.

The result: when `generate_memo()` reaches `create_artifact_directory(firm=firm)` and that calls `paths.resolve_deal_context()`, the resolver finds the config and routes the run through the firm-scoped path. Output lands at `io/{firm}/deals/{company}/outputs/{company}-{version}/` — exactly where the firm-pill in the UI implies.

When `firm` isn't provided, the scaffold is a no-op and the run still falls through to legacy `output/`. CLI users who rely on that behavior aren't disturbed.

### `tests/` — pytest is real now

`pyproject.toml` gained a `[tool.pytest.ini_options]` block (`testpaths = ["tests"]`, `asyncio_mode = "auto"`). The test suite now covers:

| Test                                                  | What it pins down                                          |
| ----------------------------------------------------- | ---------------------------------------------------------- |
| `test_create_memo_request_rejects_empty_body`         | Pydantic validation on `POST /memos` — 422 with detail.    |
| `test_get_unknown_job_returns_404`                    | Dispatcher's "job not found" shape.                        |
| `test_log_sink_splits_on_newlines`                    | Multi-line writes → one event per complete line.           |
| `test_log_sink_buffers_partial_line_until_flush`      | Trailing partial line stays buffered until flush.          |
| `test_event_bus_replays_backlog_to_late_subscribers`  | Late SSE subscribers see backlog + live tail.              |
| `test_milestone_extractor_recognizes_known_prints`    | Real-shaped agent prints map to correct stage + label.     |
| `test_milestone_extractor_dedupes_static_milestones`  | A repeat of a parameterless print doesn't re-fire.         |
| `test_milestone_extractor_ignores_unmatched_lines`    | Unmatched lines produce no milestone.                      |
| `test_scaffold_firm_deal_dir_writes_config`           | Firm-scoped POST creates `inputs/deal.json` from request.  |
| `test_scaffold_firm_deal_dir_preserves_existing_config` | Pre-existing config is not overwritten.                  |

12 tests, ~0.16s end to end. None of them spend a cent on Anthropic — `generate_memo()` is never called. They use FastAPI's `TestClient`, `pytest.tmp_path`, `monkeypatch.setenv("MEMO_IO_ROOT", ...)`, and direct unit calls into the helpers.

A small refactor was needed for the path-traversal test: the `target.relative_to(out)` logic was inline in the `get_artifact` endpoint. It's now a module-level `resolve_artifact_path(out, sub)` that the endpoint calls and the test exercises directly. Cleaner factoring; identical behavior.

## Design Decisions

**Pattern matching, not orchestrator instrumentation.** The agents could have been changed to emit explicit "stage start / stage complete" events. They weren't. Instead, the sidecar pattern-matches the prints the agents already produce. Reasons: (a) zero touch to the agent code keeps the CLI behavior identical, (b) the prints already exist and are well-shaped, (c) when an agent's prose changes, we update one regex in `milestones.py` rather than chasing through 30 agent files. The cost is fragility — a print rewrite breaks a milestone — but the test suite can grow with the catalog and the failure mode is graceful (you fall back to the raw log, which still works).

**Milestone events on the same bus as logs, not a separate channel.** A separate `/memos/{id}/milestones` endpoint was tempting, but the SSE bus already carries the chronological order both event types need. Clients render them where they want; both arrive on the same stream with the same `ts`.

**The scaffold writes only what the request carries.** `deal.json` is the orchestrator's contract — it accepts a wide set of optional fields. The scaffold doesn't try to invent defaults; it copies through only what the request explicitly provided. If the user later wants to enrich the file (add `stage`, edit `notes`), they can — the scaffold won't fight them.

**Idempotency over overwrite.** A user who runs the CLI to set up a deal, then later runs the API for the same firm+company, should not lose their hand-curated config. The scaffold checks for an existing `inputs/deal.json` and bails out untouched. If the API request had richer fields than the existing config, those richer fields are silently ignored — but the run still routes to the right place because `ctx.exists()` returns True, which is the only thing `resolve_deal_context()` cares about.

**Pytest config in `pyproject.toml`, not a separate `pytest.ini`.** Matches the modern Python convention and keeps the repo's tool config in one file. `asyncio_mode = "auto"` so `@pytest.mark.asyncio` decorators aren't needed on every async test.

## What's Next

The desktop client (memopop-native) has been built out to consume these events alongside this work — a fullscreen `JobView` with a phase checklist (left), the live log stream (center, dominant), and a polling artifact browser (right). That's documented in memopop-ai's own changelog; the orchestrator doesn't need to know about it.

A few orchestrator-side follow-ups worth keeping on the radar:

1. **More milestone coverage.** The current 30 patterns hit the loudest agent prints. Smaller agents (citation_validator, fact_corrector per-section progress, scorecard per-dimension scoring) still flow only through raw logs. Adding patterns is one-line each.
2. **Estimated time remaining.** A simple model — average duration per stage observed across recent runs — could let the API surface an ETA on `GET /memos/{id}`. Useful for the desktop client to render "≈ 12 min remaining."
3. **Resume from artifact tree.** If a run crashed mid-flight (the desktop client closing a window currently kills the sidecar process via `WindowEvent::Destroyed`), `state.json` is on disk and `python -m src.main --resume` works. The API doesn't expose a resume verb yet. `POST /memos/{id}/resume` would be a short follow-up.
4. **Persistent job history.** Process memory is the store today; restarting the server loses the in-flight registry. SQLite-backed persistence is a 2-hour follow-up if/when the use case justifies it.

None of these block the current loop from feeling tight.
