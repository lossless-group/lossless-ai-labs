## Why Care?

If you've ever pointed an AI tool at a spreadsheet of companies or contacts
and asked it to enrich them, you know the failure mode: a bulk run that goes
sideways, fills the wrong columns, hallucinates LinkedIn URLs, or quietly
returns nothing for half the rows. The fix isn't a smarter model — it's a
**gate**. Something has to sit between "I want this" and "the records change."

augment-it's gate is conversational. You state the goal. The chat drafts the
prompt. You react to the draft. The chat refines it. You explicitly click
run. Then — and only then — the records change. The same chat surface
watches the enrichment finish and checks whether the prompt actually
delivered what it promised.

That's the entire v0.0.1 thesis: the chat is the place where the user and
the model **author the enrichment together**, before any records move.

## What's New?

- A chat panel ships in the augment-it shell — Configuration A from
  [[Slides_Anatomy-of-the-In-App-Agent-Shell]] (chat-as-primary).
- Four new capabilities in `@augment-it/workspace`:
  - `prompts.draft` — drafts a prompt template from a user goal + a sample
    of the record set's columns
  - `prompts.improve` — given a draft + feedback, produces a refined version
    linked to its parent via `derived_from`
  - `prompts.apply` — a ScriptCapability that binds the drafted prompt to a
    record set scope and runs the enrichment, with **postconditions** that
    check the result (rows enriched, exit code, no `rows_enriched=0`)
  - `records.list` — read the records as they look right now
- A new `Prompt` entity in the workspace with draft-versioning (`derived_from`
  preserves the iteration history; `improve` creates a new version, never
  mutates in place).
- A chat service container (`services/chat/`) — Node/Fastify, SSE,
  assembles the four-slab system prompt with explicit cache breakpoints.
- A federation remote for the chat surface (`apps/chat/`) with four
  components, the load-bearing one being `PromptDraftPanel.svelte` —
  drafts render **inline in the chat conversation**, not in a side panel.
- An anticipation map keyed off `(activeView.kind, last_capability)` — the
  chat suggests 1–3 next verbs every time a capability completes.
- Strict alignment mode by default: the model is biased toward `propose`
  over `invoke` for anything ambiguous. Good demo posture; live-mode
  tightening is a v0.0.2 knob.

## How it Works

match what actually shipped. -->

Four capabilities, two adapter shapes ([[Chat-As-Verb-Surface-Patterns]]
Pattern 1): three TypeScript handlers and one ScriptCapability with
postconditions.

```
user types goal
   ↓
prompts.draft  ────────────────────────────────────►  draft renders inline
   ↓                                                       ↓
   workspace.prompts.save({ status: 'draft' })             ↓
                                                           ↓
                                              user reacts conversationally
                                                           ↓
prompts.improve  ───────────────────────────────►  refined draft renders inline
   ↓                                                       ↓
   new Prompt with derived_from = parent.id                ↓
                                                           ↓
                                                    user clicks "Run this"
                                                           ↓
prompts.apply  ────────────────────────────────►  character-cast row narrates
   ↓                                                       ↓
   ScriptCapability fires                                  ↓
   postconditions evaluate                                 ↓
   - exit_code == 0                                        ↓
   - rows_enriched >= 1                                    ↓
   - stdout contains rows_enriched=N (and not =0)         ↓
                                                           ↓
                                              anticipation suggests records.list
```

The chat is the only thing that talks to the model. The capability registry
is the only thing that mutates the workspace. Drafts are inert — they live
in the workspace as Prompt entities but no record is touched until `apply`
fires. This is the gating discipline from
[[project_augment_it_gating_and_microfrontend_thesis]] rendered as a verb
triad.

### The three response modes

The chat's static system-prompt slab declares three intents the model
chooses between every turn:

- `answer` — text only, no capability invoked
- `propose` — one or more capability suggestions with prefilled args,
  awaiting user accept
- `invoke` — tool call against a single capability

For v0.0.1, default alignment is **strict**: when the user's intent is
ambiguous, the chat proposes rather than invokes. The user clicks once
more to confirm. For a client demo this reads as the chat asking before
acting — exactly the posture you want.

The system prompt has **four cache-eligible slabs** in stable order
([[Chat-As-Verb-Surface-Patterns]] Pattern 5):

1. Static spine (package-version-pinned framing + the three response modes
   + one explicit instruction that `improve` doesn't `apply`)
2. Capability schemas for the per-turn whitelist
3. Active skills — empty in v0.0.1
4. Per-org reminders — empty in v0.0.1

Cache breakpoints sit between each slab. Even with slabs 3 and 4 empty for
v0.0.1, the breakpoints are scaffolded so v0.0.2 doesn't have to
restructure.

The `prompts.apply` ScriptCapability's `expected_effect.postconditions`
is the new quality-monitoring surface. Two `stdout_matches` postconditions
together encode "we got an explicit row count, and it wasn't zero" — a
`rows_enriched=0` outcome is treated as a violation, not a successful
no-op. That's the failure shape the gated-enhancement pattern is meant to
catch.

## What's Next

- **McpCapability adapter** — likely firecrawl or tavily wrapped as
  `corpus.search` or similar, so the chat can pull live company-page
  data during `prompts.apply` instead of relying on the model's training
  data alone.
- **SkillCapability adapter** — wrap `crawl-fetch-ingest` so the chat can
  plan an ingestion run from the SKILL.md trigger pattern, then `apply`
  executes it.
- **`requires_user_confirmation` gating** on `prompts.apply` for non-trivial
  scopes (anything over ~10 records). v0.0.1 ran without it because the
  demo arc *was* the confirmation.
- **Per-organization reminders + AI memory cache** — the `reminders.*` and
  `cache.*` capabilities from [[Memory-Layers-for-the-In-App-Chat-Package]]
  once the `studies/memory-layers-for-agents/neo` reads land.
- **Memopop adopts the patterns next**, then dididecks. The shared
  `@lossless/in-app-agent` package extraction happens after the second
  app, not the first — wait for the abstraction to be honest.
