## Why Care?

DidiDecks now has three clients in flight — calmstorm-decks, chroma-decks, humain-vc-decks — and onboarding the third surfaced a problem the first two never caught: **the shared shell wasn't actually that shared**. Several deck-generic primitives — scroll-snap pagination, arrow-key navigation, the section counter, the per-slide shell wrapper, the three-mode site toggle and its runtime, the registry-derivation utility that powers landing dashboards — had all been built inside `client-sites/chroma-decks/src/` *before* `apps/deck-shell/` was carved out, and never got lifted. As long as chroma was the only client-site, this was invisible. Onboarding humain made it visible the moment the dev server booted: the variants rendered, but there was no keyboard nav, no scroll-snap, no section counter, and the mode toggle clicked silently with no repaint.

Six components, one runtime, one lib, and one architectural fix later, every one of those gaps is closed and the shell finally is what it always claimed to be — the single home for behaviors that belong to "the deck" rather than "this client's branding." The next client onboards into a much shorter scaffolding sequence and inherits all six primitives for free. Chroma still runs against its local copies and is uninterrupted; its migration is held until humain validates the lifted versions in browser.

The practical upshot:

- **A third client is in the tree.** humain-vc-decks (`https://github.com/lossless-group/humain-vc-decks`) — private repo, tracking `development`, full DESIGN.md derived from humain.vc's production CSS bundle, three competing scroll-deck variants (`proto` / `tech-bio-canon` / `lab-notebook`), 29 slots each, content sourced from one canonical narrative file (`context-v/narratives/Humain-Deck-Content.md`) per chroma-style discipline.
- **Six primitives lifted into the shell.** PageAsDeckWrapper (scroll-snap + keyboard contract), SlideShell (per-slide wrapper with consumer-pluggable mark slot), ModeToggle + mode-switcher (per-client localStorage namespace), SlideStatusMatrix (sibling to existing DeckMatrix; reads DeckOverview), deck-overview lib. All ddd-prefixed CSS, all themable via `--ddd-*` token fallback chains, all importable as `@dididecks/shell/{components,runtime,lib}/*`.
- **Three-mode theme architecture got an invariant.** humain's first cut of `theme.css` had empty `[data-mode="dark"]` and `[data-mode="vibrant"]` blocks; the toggle visibly did nothing because `:root` carried the light values and absent declarations don't override. Fixed in theme.css, codified in humain's DESIGN.md as a frontmatter `modes:` block + a load-bearing "Architectural invariant" prose section, and saved to user-level memory as `feedback_three-mode-theme-architecture` so the same bug can't ship on the fourth client.

## What's New?

### At the deck-operating-system tier

- **`@dididecks/shell` now ships scroll-deck pagination.** `apps/deck-shell/src/components/PageAsDeckWrapper.astro` — promoted from `chroma-decks/src/layouts/PageAsDeckWrapper.astro` (commit `c87c643`). Delivers `scroll-snap-type: y mandatory`, the full keyboard contract (`↑/↓`, `PageUp/Down`, `Home`, `End`, `c` chrome toggle, `f` fullscreen), double-click step, the floating `NN / total` section indicator, `#s-N` hash-jump on load, `.reveal-item` IntersectionObserver, and the `deck:section-changed` / `deck:section-prev` / `deck:section-next` event protocol that lets `DeckChrome` buttons drive the same nav as keyboard / scroll. CSS classes prefixed `ddd-*`; theming via `--ddd-scroll-*` tokens with fallbacks so consumers aren't required to declare them.

- **`@dididecks/shell` now ships the per-slide wrapper.** `apps/deck-shell/src/components/SlideShell.astro` — promoted from chroma. The original hard-coded `<ChromaMark>`; the shell version exposes `<slot name="mark">` so consumers plug their own wordmark (or omit it). Same `ddd-slide-*` class prefix discipline and `--ddd-slide-*` token fallback chain.

- **`@dididecks/shell` now ships the three-mode toggle + its runtime.** `apps/deck-shell/src/components/ModeToggle.astro` + `apps/deck-shell/src/runtime/mode-switcher.ts` — promoted from chroma's `src/components/basics/ModeToggle.astro` + `src/utils/mode-switcher.js`. Rewritten as a TS factory `createModeSwitcher({ client, defaultMode, respectSystemPreference })`. The `client` argument becomes the localStorage namespace (`{client}:mode`) so multiple decks on the same domain don't collide on mode persistence. `defaultMode` and `respectSystemPreference` are per-client knobs so a light-canon brand doesn't get a dark first-visit just because the OS is in dark mode.

- **`@dididecks/shell` now ships the deck-overview lib.** `apps/deck-shell/src/lib/deck-overview.ts` — promoted from chroma's `src/lib/deck-overview.ts`. Reads `decksRegistryPath` + `slidesComponentsRoot` from the resolved shell options on `globalThis.__dididecksShellOptions` instead of importing the consumer's TS registries directly, so any client-site can call `loadDeckOverview("pitch")` without the shell knowing the consumer's import paths. Chroma's substantiation counts (people, headshots, investor firms, portfolio companies) were dropped from the shell version — those are per-client substantiation layers.

- **`@dididecks/shell` now ships the build-status matrix as a sibling to DeckMatrix.** `apps/deck-shell/src/components/SlideStatusMatrix.astro` — promoted from chroma. Lightweight slot × variant rollup showing static-port readiness via glyphs (`●` static / `◐` legacy / `◯` registered / `·` absent). Reads the `DeckOverview` shape produced by `lib/deck-overview`. Distinct from `DeckMatrix` (which is the production audit/review surface — dual-surface ratings, TOC integration); they're siblings, not duplicates, and the plan documents which is which.

- **`@dididecks/shell` package exports `./lib/*`.** Added to `apps/deck-shell/package.json` so `import { loadDeckOverview } from "@dididecks/shell/lib/deck-overview"` resolves. Existing `./components/*` and `./runtime/*` exports were already there.

- **`pnpm-workspace.yaml` includes humain-vc-decks.** Workspace registration so `@dididecks/shell: "workspace:*"` resolves on humain's side without copying the shell into the submodule.

- **A 5-phase plan documents the lift backlog.** `context-v/plans/Lift-Chroma-Decks-Generic-Code-into-Shared-Shell.md` — names the bug ("chroma was built before the shell carve-out; not everything got lifted"), enumerates the audit list, scopes each phase's risk + acceptance criteria, and explicitly defers the chroma migration (Phase 3) until humain's consumption of the lifted versions is verified in browser. Phases 1, 2, 4, 5 shipped tonight; Phase 3 is held.

### Inside humain-vc-decks (the substantive engagement install)

- **Repo created and registered as a submodule.** `https://github.com/lossless-group/humain-vc-decks` — private, default branch `development`, registered in the parent's `.gitmodules` and `pnpm-workspace.yaml`.

- **DESIGN.md derived from humain.vc's production CSS.** Vite-built SPA; extracted the CSS bundle (`/assets/index-zwbOMG5o.css`), pulled the shadcn-style HSL tokens, hex-equivalents-ed them into Tier-1 named tokens, mapped semantic Tier-2, recorded the source URLs + fetch timestamp + extraction notes in `context-v/explorations/Humain-Brand-and-Deck-Notes.md`. Two-color identity captured: deep-navy `#0a1326` + bio-green `#29a380`, plus pale-green accent. Typography: Space Grotesk (display) + Inter (body). Brand voice: TechBio-credible, pull-quote-forward.

- **Three competing scroll-deck variants authored end-to-end.** All 29 slots, all three variants:
  - **`proto`** — faithful PDF recreation. White surfaces, gray text, hairline borders. The baseline.
  - **`tech-bio-canon`** — humain.vc visual vocabulary applied literally. Deep-navy gradient heroes alternating with off-white card sections, bio-green eyebrows, Space Grotesk display, pull-quote treatment on the Schmidt/Huang/Jobs chorus.
  - **`lab-notebook`** — scientific-editorial direction. Monospace breadcrumb headers `humain-ventures / pitch / lab-notebook / slot-NN`, `§N` section numbers, hairline rules, `dl/dt/dd` metadata, ASCII bio-pipeline diagram on the architecture slot, bio-green as margin-rule color only.

- **Content sourced from one canonical markdown file.** `context-v/narratives/Humain-Deck-Content.md` — 29 slots, PDF-verbatim. The three variant `.astro` files consume from it; they diverge in presentation, not content. Matches chroma's `context-v/narratives/slides/` discipline (one file vs. 29 small files chosen for initial-pass efficiency; can split per-slot later).

- **Scroll-snap + keyboard nav landed via shell import.** All three variant `.astro` files wrap their sections in `<PageAsDeckWrapper>` imported from `@dididecks/shell/components/PageAsDeckWrapper.astro`. The narrow editorial column in `lab-notebook` is preserved via a scoped `<style>` rule on `section[data-variant="lab-notebook"]` (the old `max-w-4xl mx-auto` lived on the now-removed `<main>`).

- **Landing page wires ModeToggle + SlideStatusMatrix + loadDeckOverview from the shell.** First fresh consumer of those three primitives — validates that the lifts work for a greenfield client that never had local copies to fall back on.

- **Three-mode theme architecture fixed and documented as an invariant.** The original `theme.css` had `:root, [data-mode="light"] { /* full mapping */ }` paired with `[data-mode="dark"] {}` and `[data-mode="vibrant"] {}` empty stubs. The toggle was running but `:root`'s light values won by default and no Tier-2 token actually changed. Refactored to three parallel mode rules — same keyset, different Tier-1 references per mode. Added Tier-1 tokens that dark/vibrant required (`humain_void_black`, `humain_bio_green_vivid`, `humain_border_vivid`, etc.). DESIGN.md now carries a structured `modes:` frontmatter block specifying the per-mode Tier-2 + `--fx-*` mappings; the `theme.css` runtime mirrors that block 1:1. A new "Architectural invariant" prose section names the bug and the rule for future-Claude. Saved to `~/.claude` memory as `feedback_three-mode-theme-architecture` so the invariant is enforced on every future client-site onboarding, not just this fix.

- **Auth installation is its own next plan.** `client-sites/humain-vc-decks/context-v/plans/Install-Auth-Surface-from-Calmstorm-Pattern.md` — full port path from chroma's auth surface (middleware + astro:db + arctic OAuth + magic-link + passcode), with `humain.vc` seeded alongside `lossless.group` as orgs per the domain-as-id convention. Held until a deck variant is presentable enough to gate.

## What's Held

When unblocked, the cleanest path is thin shim re-exports at chroma's original local paths (`src/layouts/PageAsDeckWrapper.astro`, `src/utils/mode-switcher.js`, etc.) so chroma's 20+ files that import those paths don't need updating; only the implementation comes from the shell. Local files get deleted only after parity is verified end-to-end.
