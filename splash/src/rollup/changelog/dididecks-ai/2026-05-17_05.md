# /data-assets/companies and /data-assets/people shell-injected; DeckStatsPanel lifts the homegrown chroma stats tile into shared shell ground

## Why care?

The goal stated explicitly this evening: **move everything but the underlying content and data to the shell**. The shell should provide the UI; consumer sites should provide the deck content and the firm/people/company .md records. After today's lift, every UI surface for the reviewer audit ŌĆö the stats tile row on the home page, the per-firm grouped companies table, the role-grouped people table ŌĆö lives in `@dididecks/shell`. Chroma's `pages/index.astro` is now 100 lines lighter and stops carrying logic that belongs at the shell tier.

The four-tile row was working visually but broken functionally: People and Companies tiles linked to `/data/team/` and `/data/investors/`, neither of which was a route in chroma ŌĆö they were never wired up after the calmstorm-style page was sketched. Today's work moves the destinations into existence (as shell-injected routes) and points the tiles at them.

## The story

### Two layouts, one shell

Calmstorm-decks puts a single firm's data at `data/firms/calm-storm-ventures/{portfolio,team}/*.md`. Chroma-decks puts four investor firms' portfolios at `data/investors/{firm}/portfolio/*.md` and people flat at `data/team/*.md`. Same conceptual data, different file layouts. The first instinct was to add a `dataAssets` config option to the shell integration so each consumer could declare its own glob patterns. Rejected after one minute's thought: `import.meta.glob` requires literal string arguments at Vite's parse stage, not runtime config. Glob patterns can't be configured per consumer; they have to be hard-coded.

The fix that worked: pick glob patterns that cover both layouts simultaneously.

- Companies: `/data/**/portfolio/*.md` catches `data/firms/.../portfolio/` AND `data/investors/.../portfolio/` AND any future shape that puts portfolio under `data/`.
- People: `/data/**/{team,portfolio}/*.md` with a `role_class` frontmatter filter catches the cross-layout reality ŌĆö calmstorm puts portco-CEOs under `portfolio/`, chroma puts everyone under `team/`. The role_class filter is the structural test, not the path.
- Firm grouping: path-based extraction. `data/firms/{firm}/portfolio/` and `data/investors/{firm}/portfolio/` both surface the firm slug at the same path position; a tiny `firmSlugFromPath` regex (later string-method'd, see traps below) handles both.

The result: zero config knobs on the shell side. The two consumers shipping today and any third consumer that follows the same `data/(any)/(portfolio|team)/` convention gets the audit pages for free.

### The three Astro-pipeline traps

**Trap 1 ŌĆö JSDoc terminator inside path documentation.** The people route's header comment originally read:

```
 * One row per person .md found by the glob. "Person" = any .md under
 * /data/**/team/ OR /data/**/portfolio/ that carries a `role_class`
 * frontmatter field.
```

The `/data/**/team/` contains the sequence `**/` ŌĆö and `*/` ends a JSDoc block. The comment terminated early on line 6; the rest was parsed as code; build failed with "Expected ; but found portfolio" at line 6 col 30. **Fix**: rewrite path examples in prose inside JSDoc, or wrap them in backticks.

**Trap 2 ŌĆö Inline `/* */` block comment misparsed as regex literal.** `DeckStatsPanel.astro` originally had:

```ts
const staticPorted = authoredSlotFiles; /* every per-slide file is by definition a static-play port */
```

esbuild's TS pre-parser saw `;` and re-entered expression position. The `/*` that followed got classified as the start of a regex literal rather than a block comment. Build failed with "Unterminated regular expression" at a column position past the end of the line ŌĆö the parser was reporting where the regex *should* have terminated, not where it started. The diagnostic signature was confusing because the highlighted line was usually a one-character `}`.

A second variant of the same trap: complex multi-slash regex literals (e.g. `p.match(/^\/data\/(?:firms|investors)\/([^/]+)\/portfolio\//)`) in expression-adjacent positions can trip the same regex-vs-divide ambiguity, especially when chained with `??`. The patterns themselves are valid; the parser's classification of the surrounding context is the issue.

**Fix**: line comments over inline block comments; string methods (`split`, `startsWith`, `endsWith`, `lastIndexOf`) over slash-heavy regex literals when the logic is amenable; if regex is genuinely the right tool, hoist it to a top-of-file `const`.

Both lessons captured in the new `astro-knots/context-v/reminders/Astro-Frontmatter-Avoid-Inline-Block-Comments-and-Risky-Regex.md` reminder.

**Trap 3 ŌĆö Astro `<Image>` over consumer data is fragile.** The first companies-page build failed on `[NoImageMetadata] Could not load .../trademark__V7.svg`. One corrupted SVG in chroma's 373-row data tree was enough to take down the page render. Astro's `<Image>` component runs metadata parsing on every asset; a malformed file in *any* asset blows the whole page. For audit pages over raw consumer-dropped data ŌĆö where the whole point is to surface inconsistencies ŌĆö `<Image>` is exactly the wrong choice.

**Fix**: use `import.meta.glob` with `{ query: "?url", import: "default" }` to get plain URL strings, and render via raw `<img src>`. Skips the metadata pipeline entirely; one corrupted SVG renders as a broken image icon instead of killing the page. Worth a follow-up reminder if this recurs.

## How it works (or "Under the hood")

### Layout-agnostic globs via `**` zero-or-more matching

Vite's `import.meta.glob` (under fast-glob) supports `**` as "match zero or more path components." That property is what lets one glob pattern cover both layouts:

```ts
// Catches:
//   /data/firms/calm-storm-ventures/portfolio/foo.md       (calmstorm ŌĆö depth 4)
//   /data/investors/quiet-capital/portfolio/anduril.md     (chroma     ŌĆö depth 4)
//   /data/portfolio/foo.md                                 (hypothetical ŌĆö depth 2)
const mdGlob = import.meta.glob<string>(
  "/data/**/portfolio/*.md",
  { eager: true, query: "?raw", import: "default" },
);
```

For people, the union pattern `{team,portfolio}` plus the `role_class` content filter means a single glob covers calmstorm's per-firm team dir, chroma's flat team dir, and calmstorm's portco-CEO .md files that live under `portfolio/`. The filter happens in-component, not in the glob, so the structural test for personhood is data-driven (presence of `role_class`) rather than path-driven (which dir it's in).

### Why `<DeckStatsPanel>` does its own counting

The local `AssetsDataPanel` in chroma was fed by a `loadDeckOverview` library that wrapped the same registry/glob calls. Two reasons not to take that library into the shell:

1. The shell already has direct access to `loadDecksRegistry`, `loadSlotsRegistry`, and `perSlideFileExists`. Wrapping those in a separate overview library is one layer of abstraction the shell doesn't need ŌĆö the panel can do its own counting in 30 lines.
2. The panel's data should always match the audit pages exactly. Having the panel and the audit page read from the same globs guarantees consistency; routing both through an intermediate library is a place where the two views could quietly diverge.

The chroma-side `loadDeckOverview` still exists because the variant cards above the matrix consume it. When those variant cards migrate to a shell component (likely a future iteration), the local library can retire.

### Resilient image rendering for audit pages

```ts
// Note: NOT { eager: true } returning ImageMetadata. Plain URL imports:
const imageGlob = import.meta.glob<string>(
  "/data/**/portfolio/*.{png,jpg,jpeg,webp,avif,svg}",
  { eager: true, query: "?url", import: "default" },
);

// In the template, plain <img> instead of Astro's <Image>:
<img src={logo} alt={`${c.name} logo`} loading="lazy" />
```

Astro's `<Image>` component is the right choice for end-user-facing pages where optimization matters and the asset set is curated. For reviewer audit pages over consumer-dropped data, the audit is exactly the surface that *should* be able to render a broken thumbnail rather than 500 the whole page ŌĆö that's how the reviewer learns there's a broken file to investigate.
