## Why Care?

If you'd tried to share a chroma deck URL with an investor between May 17 and tonight, they'd have hit the access gate, entered their passcode (good), been redirected to the deck URL (good), and immediately seen a server-error page reading `Cannot read properties of undefined (reading 'length')` instead of slide 10 of the pitch deck (very not good). The auth gate worked. The deck on the other side of the gate wasn't loadable. We caught it because we were re-running the end-to-end auth flow before bundling the next batch of work.

The fix is a six-line restructure in two shell route files. The story is mostly about what *not* to fix it with — the wrong fix was right there, ready to ship, and would have re-opened the exact prerender-bypass leak we closed nine days ago.

## What's New?

- Two shell routes (`@dididecks/shell`'s `play/[deck]/[variant]/[slot].astro` and `play/[deck]/[variant]/print.astro`) no longer depend on `Astro.props` for per-route data. They load the deck + slot registries inline from `Astro.params` at request time.
- Same render path in static and SSR builds — works for chroma (`output: "server"`, no prerender, middleware gates everything) AND for any future static consumer.
- Verified end-to-end via curl: `POST /api/access/verify` (admin passcode) → 302 + `cd_session` cookie → `GET /play/pitch/enhanced-v2/10/` with that cookie → `HTTP 200 OK`, response header `x-cd-auth: ok-org_admin`, ~102 KB of slide HTML.

## What was actually broken

The shell's `[slot].astro` frontmatter started, roughly, like this:

```ts
const { slug, title, slotIdx, slots, variantLabel, deckVariants, variantIdx } = Astro.props as {
  slug: string;
  title: string;
  slotIdx: number;
  slots: SlotRef[];
  variantLabel: string;
  // ...
};

const totalSlots = slots.length;   // ← line 1 of the rendered route
```

`Astro.props` is populated by whatever `getStaticPaths` returned for the matched path. But in `output: "server"` mode, dynamic routes without `export const prerender = true` are **on-demand** — `getStaticPaths` is never called, and `Astro.props` is empty. `slots` is `undefined`. `slots.length` throws. Astro returns a 500.

Calmstorm doesn't hit this because calmstorm's deck pages live in `client-sites/calmstorm-decks/src/pages/play/...` — a totally different URL shape that bypasses the shell's `/play/[deckSlug]/[variantSlug]/...` injected routes entirely. **Chroma is the first consumer to compose the shell's deck routes under SSR.** Greenfield catches things older repos coast past.

## Why not just `prerender = true`?

I almost shipped this as the fix. I had the edit drafted in five places. Then I read the middleware comment chroma carries:

```ts
// IMPORTANT — prerender + middleware interaction. Under `output: "server"`
// on Vercel, pages with `export const prerender = true` are CDN-served as
// static HTML and the middleware function NEVER RUNS for them in production
// (it does in dev, with a separate cookie-visibility bug). For a gated deck
// this means anyone with a URL can play the deck — exactly what we don't
// want. Calmstorm's 2026-05-17 fix (commit 677417f) was to remove prerender
// from every gated route. Chroma followed suit: NO deck page has prerender.
// Only public surfaces (none currently) would.
```

That's the same trap we just got out of nine days ago. The `astro.config.mjs` comment in chroma was stale and suggested `prerender = true` *was* the fix; the middleware comment is the authoritative one and explicitly forbids it. **The fix has to keep the middleware in the request path on every deck visit, which means no prerender on gated routes.** Any solution that puts `export const prerender = true` on a gated deck page is wrong even if it makes the symptom go away — because the new symptom is "investor-facing material is now world-readable."

I reverted the five `prerender = true` edits I'd just made and looked for a different way.

## The actual fix

Both `[slot].astro` and `print.astro` now do this:

1. Keep `getStaticPaths` returning **only params** (no props). It still works for any future static consumer; it's just not the source of per-route data anymore.
2. In the frontmatter, call `loadDecksRegistry(opts.absolute.decksRegistry)` and `loadSlotsRegistry(opts.absolute.slotsRegistry)` **inline from `Astro.params`**.
3. Resolve `deck`, `variant`, the slot row, the slot list, `variantLabel`, `deckVariants` etc. from the loaded registry data.
4. Return a 404 `Response` if the deck/variant/slot in the URL doesn't exist (these guards used to be implicit — `getStaticPaths` couldn't produce an invalid path).

The 130+ lines of render code below the frontmatter — `prevSlot`, `nextSlot`, `firstSlot`, `lastSlot`, the variant-cycle math, the `allVariants` menu, the `intent` URL param — none of it changed. What changed is **where the values feeding it come from**: the registry, loaded per request, instead of `Astro.props`, frozen at build time.

The cost is one extra `loadDecksRegistry` + `loadSlotsRegistry` per request. Each is a `fs.readFile` + an esbuild transform — low single-digit milliseconds on a warm dev server. For a private fundraise deck with low concurrency and a small registry, irrelevant. For a future high-traffic consumer, the registry loader already caches the esbuild import; a Phase-B improvement could memoize it per process.

## A cookie red herring

While verifying the cookie flow with `curl`, I hit something that looked like a separate bug: `POST /api/access/verify` returned a valid `cd_session` cookie, but `GET /play/...` with that cookie still redirected back to `/access` *without* `&error=expired` in the URL. The middleware was acting as if no cookie had been sent — even though `curl -v` clearly showed it being sent.

I wrote a tiny diagnostic endpoint that dumped the request's cookie state through Astro's `cookies` API. Result: Astro saw the cookie correctly. Re-ran the auth flow → `200 OK`, `x-cd-auth: ok-org_admin`, deck rendered. Best read: the dev server had transient stale state from the in-progress edits, and the file watcher detecting the diagnostic-endpoint file creation kicked the module graph into picking up my fixes. The cookie API was never the problem; the route just hadn't fully reloaded yet. Diagnostic endpoint was deleted before the commit; noted here because the apparent cookie bug almost sent the investigation off course for an hour.

## Files touched

- `apps/deck-shell/src/routes/play/[deckSlug]/[variantSlug]/[slot].astro`
- `apps/deck-shell/src/routes/play/[deckSlug]/[variantSlug]/print.astro`

Paths are relative to the **dididecks-ai parent**, not chroma-decks. The chroma-decks repo carries `@dididecks/shell` via a pnpm workspace symlink (`node_modules/@dididecks/shell → ../../../../apps/deck-shell`), so the fix lands automatically — no chroma-decks source files needed to change.

The `toc.astro` and `toc-deck.astro` shell routes were already SSR-safe (they had been loading registries inline from `Astro.params` since they were written). The `/play/[deckSlug]/[variantSlug]/index.astro` redirect route uses only `Astro.params` and didn't need touching.

## What this enables next

- **Chroma's gated decks load.** Investors who get past the gate now see the deck instead of a 500.
- **Any future SSR consumer of `@dididecks/shell` works out of the box.** The shell routes no longer assume the consumer is in static mode.
- **The package-extraction path stays clean.** When `@dididecks/shell` eventually moves to its own repo, SSR-safety is table stakes, already in.
