## Why Care?

A startup raising Series A normally lives or dies by a deck that exists as a single mutable file — usually a Figma board or a Pitch.com URL the founder updates between meetings. The "current best version" is whatever was last opened. There are no branches. There are no variants. There is no substantiation layer next to the claims. There is no "this deck plays well in print, on a phone preview, and on the big screen in a board room." There is just *the file.*

Today the Chroma fundraise workspace stopped being that. **The 14 slides the CEO has been presenting live are now code** — one Astro component per slide, sharing a three-mode design system extracted directly from `trychroma.com`'s production CSS, sitting next to a private `corpus/` of MemoPop-generated investment memos and the founder's own working files. The deck is **versioned, themeable, share-preview-able, and editable in isolation**. Edit slide 7 without scrolling past slides 1–6 and 8–14. Rebuild in light mode for the deck the CEO presents from; bounce to dark or vibrant to feel which palette wants to live where next.

This is the first concrete test of the [[Dididecks-AI-Slide-Decks-as-Code]] thesis on a real engagement. The bar wasn't *make a better deck.* The bar was *match the deck the founder is already presenting from, exactly, in code, so that everything we want to do next — variants, gated publishing, hi-fi Keynote-grade export, mode-aware sharing — has a true substrate to iterate from.*

## What's New?

A coherent slab of work in one day, all on `lossless-group/chroma-decks` (private), tracking `development` → `main` → `master`:

- **Private Astro 6 / Tailwind v4 / Vercel-static scaffold** at `dididecks-ai/client-sites/chroma-decks`, attached as a submodule of the dididecks-ai pseudomonorepo with `branch = development` in `.gitmodules`. Three tier branches at parity from inaugural commit.
- **Three-modes design system** — light (Chroma's canon), dark ("warm studio" — brand black `#27201c` IS the surface), vibrant ("neon DD lab" — deeper warm shadow `#1a1410` with orange-tinted surfaces and a three-color headline gradient sweep). All three modes scaffolded from day one; rewiring tokens later causes refactor loops.
- **Two-tier token discipline** (per `theme-system` skill): 19 named Tier 1 colors (real names like `chroma-bistre`, `chroma-cinnabar`, `chroma-alabaster`, `chroma-marigold`, `chroma-cobalt` — no functional names), Tier 2 semantic tokens (`--color-background`, `--color-spotlight`, etc.) that components consume, plus a `--fx-*` effects layer that dials glow / shadow / headline-gradient intensity per mode.
- **Mode-aware Chroma trademark** — two upstream SVGs (Light-Mode and Dark-Mode trademark variants supplied by Chroma) swap via CSS based on `data-mode`. Light trademark in light, dark trademark in dark and vibrant. No JS swap, no flash.
- **`/scroll/pitch/` deck shipped with 14 component slides** recreating the live `2026-05-11_Chroma-Series-A_MS-Resort.pdf` faithfully — cover, opening, traction, competition matrix, bottleneck framing, solution stack, difficult-problems split, two-segments layout, xAI case study, team grid, business model, roadmap, ask, closing.
- **`corpus/` substantiation layer** (`.gitignore`'d, never deployed): 8 MemoPop-generated Chroma memo variants (18 MB), the management-supplied Series A PDF, the Alpha Partners investment memo, and an `aix-ventures-roster` experiment from before `crawl-fetch-ingest` was a skill. Slides cite by filename reference, not by commit.
- **`.gitattributes` for design binaries** — `.afdesign`, `.afphoto`, `.sketch`, `.fig`, `.psd`, `.ai` all marked `binary` so git never tries to text-diff them.
- **Mode toggle** in the top-right of every page (sun / moon / star), pure CSS icon swap, `localStorage` persistence, `prefers-color-scheme` first-visit default.

## The arc

It went in five beats, each one informing the next.

### 1. Scaffold from substrate, not from scratch

The instinct was to copy-paste from `calmstorm-decks` — the most-evolved client-site we've shipped. The compromise was to **rebuild fresh using calmstorm's structural ideas** (flat Astro project, `/scroll` + `/play` format namespaces, `PageAsDeckWrapper` scroll-snap primitive, Stitch-spec `DESIGN.md`, `context-v/` + `changelog/` at the root, `corpus/` git-ignored). Calmstorm-specific routes (`/thesis`), Material/M3 token names (`--color-on-surface-*`), and the gating logic all left behind. The new `/scroll` + `/play` are **thumb-gallery indexes** — they list available decks via `src/data/decks.ts` and link out to `/scroll/{slug}/`, not auto-launch one canonical slideshow.

### 2. Lock the three modes before they fight back

The temptation in light-mode-only brands is to ship light first and "add modes later." That's the trap. Every component you write referencing `var(--color-background)` becomes a *retrofit target* when you finally introduce a second mode. Tokens scaffolded from day one cost nothing extra; tokens scaffolded after the fact cost everything you've already built.

The skill canon was explicit on one anti-pattern that would have bitten us:

> **"Vibrant mode is dark-based (like dark mode, not light mode). Common error: vibrant inherits light mode's white background."**

So vibrant became a *darker* base than dark — `#1a1410` (warm-shadow) instead of `#27201c` (chroma-black) — with surfaces visibly orange-tinted via `color-mix(in srgb, var(--color__chroma-orange) 14%, var(--color__chroma-shadow))`. Headlines run a **three-color amber → orange → teal sweep** via `--fx-headline-gradient` and `background-clip: text`. If vibrant feels merely *darker than dark*, it has failed. It should feel **louder**.

### 3. Find the leaks. Fix them. Stay honest.

Half-way through, we audited what was actually in the tree and found **calmstorm-style token names had leaked** into the inherited `PageAsDeckWrapper.astro`: `--color-outline`, `--color-on-surface-muted`, `--color-on-surface-strong`, `--rounded-sm` — all of which are calmstorm's Material-flavored vocabulary and **none of which exist in chroma-decks's token system**. Plus two raw `rgb(255 255 255 / 0.9)` values — translucent white backdrops that looked correct in light mode but wrong on dark and vibrant.

Both classes of issue break the two-tier discipline: **components must read only Tier 2 semantic tokens, never raw values, and every semantic token they read must be defined in `src/styles/theme.css` and reference Tier 1 named tokens.**

Fix: introduced a per-mode `--color-glass-bg` semantic token derived from Tier 1 via `color-mix`:

```css
[data-mode="light"]   { --color-glass-bg: color-mix(in srgb, var(--color__white)        90%, transparent); }
[data-mode="dark"]    { --color-glass-bg: color-mix(in srgb, var(--color__chroma-black) 85%, transparent); }
[data-mode="vibrant"] { --color-glass-bg: color-mix(in srgb, var(--color__chroma-shadow) 88%, transparent); }
```

Then renamed five role-flavored Tier 1 tokens to **actual color names** — Tier 1 should be "things we have," not "roles":

| Was (role) | Now (real color name) | Hex |
|---|---|---|
| `chroma-border-warm` | `chroma-bistre`     | `#3a312b` |
| `chroma-cream-muted` | `chroma-oat`        | `#d4cfc2` |
| `chroma-orange-soft` | `chroma-marigold`   | `#f99c00` |
| `sidebar-white`      | `chroma-alabaster`  | `#fafafa` |
| `destructive`        | `chroma-cinnabar`   | `#df2225` |

Post-fix audit: zero raw hex / rgb in `src/`, zero undefined CSS variable references, zero semantic-token definitions outside `theme.css`.

### 4. Read the PDF. Recreate the slides.

Then the real work — 14 slides, recreated as components, in one pass.

```
src/components/slides/pitch/
├── 01-cover.astro          "Context and Memory Infrastructure for AI"
├── 02-opening.astro        "Agents will automate knowledge work…"
├── 03-traction.astro       27k stars · 14M downloads · 50K teams · $2.4M
├── 04-competition.astro    2×2 matrix · Chroma top-right · "How we win"
├── 05-bottleneck.astro     "…they are context and memory failures."
├── 06-solution.astro       5-layer Chroma stack with side annotations
├── 07-difficult.astro      Stack + system-problem / data-problem bullets
├── 08-segments.astro       Context for Teams ↔ Context for Developers
├── 09-case-study.astro     xAI / Grok + "we've saturated our existing evals"
├── 10-team.astro           12 cards (initials placeholders for now)
├── 11-business-model.astro Open-source · Cloud · Enterprise
├── 12-roadmap.astro        Sync · Database (250M → 100B) · Agent (400 → 10k)
├── 13-ask.astro            "$12M" + Chroma today + Milestones by Series B
└── 14-closing.astro        "Every agent needs context and memory"
```

A `SlideShell` layout handles shared chrome (padding, min-height, scroll-snap, optional top-right `ChromaMark`). Each slide is a self-contained ~30–120 lines. Reorder by rearranging imports in `scroll/pitch/index.astro`. Edit slide 7 without scrolling past 6 others.

**One new brand color** was needed: the **vivid blue** the deck uses to highlight "Memory & Context" / "The missing piece" / "Context & memory" boxes (slides 5, 7). It's not in trychroma.com's production CSS — it's a deck-specific accent. Added `--color__chroma-cobalt: #1a73e8` (Material/Google Blue) to Tier 1 with `--color-spotlight` Tier 2 mapping. Picked by eye from the PDF; the founder will tell us if the exact hex differs.

### 5. The honest gap

Some elements need source assets we don't have yet. They render gracefully as placeholders today, and they're labeled drop-in replacements — the layout doesn't shift when real assets land:

| Where | Placeholder today | Real asset later |
|---|---|---|
| Slide 1 + 14 — pixel-art icons (boxes / brush / terminal / airplane) | SVG approximations using `chroma-amber`, `chroma-cobalt`, `chroma-teal`, `chroma-cinnabar` brand colors | Chroma's actual pixel-art icons (drop-in same dimensions) |
| Slide 3 — 13 customer logos (xAI, Mintlify, Qualcomm, Skydance, KREA, …) | Text labels styled as dashed cards | Real SVG logos via the `crawl-fetch-ingest` skill |
| Slide 9 — xAI mark | Text "xAI" | xAI SVG |
| Slide 10 — 12 team headshots | Initials in spotlight-amber gradient circles | Real headshots, also via `crawl-fetch-ingest` |
