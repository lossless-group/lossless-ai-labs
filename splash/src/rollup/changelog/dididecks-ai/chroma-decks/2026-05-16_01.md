# Enhanced-v2 ships all 17 play slides + Blue Ocean canvas + a working /play chooser

## Why care?

If you're the founder reading this deck, the practical change since 2026-05-12 is: open `/play/`, click the card, the deck plays. All three substantive variants — `proto` (13 slots), `enhanced-v2` (17 slots), `enhanced-v3` (16 slots) — are reachable in Play-UI with arrow-key navigation. The enhanced-v2 variant got a Blue Ocean Strategy Canvas slide added (slot 09b) and the Use-of-Funds slide renamed to Use-of-Proceeds, both with the founder's editorial eye. `/print/` exports a clean PDF for sending. None of these were true on 2026-05-12.

If you're an agent landing in chroma cold, the structural change you should know about: `format: "scroll"` on Deck entries is gone. The `/play/` chooser used to filter by `decksByFormat("play")`, which always returned `[]`, hence the empty state. The corrected model: a deck **has** Play-UI iff per-slide files exist at `src/components/slides/{variant}/{slot}-{slug}.astro`. The new `src/data/play-availability.ts` discovers this at build time via `import.meta.glob`, AND-intersects with `src/data/slides.ts` `SLOTS[variant]` (the shell's authoritative source), and surfaces only `(deck, variant)` pairs where the chooser can link to a route the shell will actually emit. The chooser literally cannot send you to a 404 anymore.

## What's new

### Enhanced-v2 — 17 play slides shipped + two editorial changes
- **All 16 v2 sections ported to static 16:9 per-slide files** (`src/components/slides/enhanced-v2/01-cover.astro` through `16-closing.astro`). Per the Refactor-Scroll-Ported-Slides discipline: dropped `vw/vh/clamp` viewport units, removed IntersectionObserver-driven reveals, eliminated marquees and transitions, removed breakpoint media queries. The result renders predictably inside SlideCanvas's 1920×1080 stage and exports cleanly to PDF.
- **Slot 09b — Blue Ocean Strategy Canvas** added between 09 (Competition) and 10 (Team). ERRC framework + value curves rendered as a dense diagrammatic slide. The "b" suffix on the slot number is deliberate — slot numbers can be non-numeric in SLOTS to allow insertion between existing slots without renumbering downstream.
- **Slot 14 — Use of Proceeds** renamed from "Use of Funds" with a milestone-anchored treatment ("$10M run rate by Q4 2026"). The founder's edit; reflected in both the scroll deck and the per-slide play file.
- **Slot 15 — The Ask redesign** (mega-numeral `$12M` hero, two-column Round/Status `<dl>` blocks, no inline `<pre>` blocks). Authored as the deliberate "Play-UI can render better than Scroll-UI" exemplar.

### Deck dashboard at the variant root
- `/scroll/pitch/enhanced-v2/` got a deck-dashboard surface for the variant — investor-attribution display, a PDF-download link, and audit visibility — making the variant root useful as a shareable landing page for the founder's pre-meeting prep. The first PDF export shipped from this dashboard (`fa7d287`).

### `/play/` chooser rebuild — runtime detection
- **`DeckFormat` type retired** from `src/data/decks.ts`. The `format: "scroll" | "play"` field on Deck records is gone; `decksByFormat()` helper removed.
- **`src/data/play-availability.ts`** (new) carries the runtime detection logic. Uses `import.meta.glob` over `src/components/slides/**/*.astro` to discover per-slide files at build time. Lives *outside* `decks.ts` on purpose: the `@dididecks/shell` integration evaluates `decks.ts` via an esbuild-backed data-URL pathway where `import.meta.glob` (a Vite-only transform) is undefined and crashes the build. Splitting it into a sibling file keeps the shell's registry-loader happy.
- **AND-intersect with SLOTS.** `playableSlotsByVariant()` filters by **both** file existence (the chooser's truth source) **and** `SLOTS[variant]` membership (the shell's `getStaticPaths` truth source). The intersection guarantees the chooser can never link to a route the shell won't emit. Either source alone is misleading: a file without SLOTS isn't renderable by the shell; a SLOTS entry without a file renders the DecomposeFirstPlaceholder, which isn't "playable" in the chooser's sense.
- **Newest-variant CTA.** The card's primary link points at `variantsWithSlides[length-1]` — `deck.variants` is ordered baseline-first → newest-last, so the most-recent variant is what you want by default. The card surfaces "opens Enhanced-v3 · slot 01" so the destination isn't a surprise; "Also in Play-UI: proto, enhanced-v2" is demoted to a footnote.

### `SLOTS["proto"]` populated
- 13 entries matching the existing per-slide files at `src/components/slides/proto/` (slots 01–05 and 07–14; slot 06 deliberately skipped — proto's original 13-slide structure omits it). Without these entries the shell's `getStaticPaths` couldn't emit `/play/pitch/proto/[slot]/` pages, so the proto deck rendered with empty props in dev and 404'd cleanly in build. Now `/play/pitch/proto/01/` plays end-to-end with arrow-key nav across all 13 slots.

### `/scroll/` chooser cleanup
- Dropped the `decksByFormat("scroll")` filter — every deck has Scroll-UI by default (the long-page sections live under `src/pages/scroll/{slug}/{variant}/index.astro`), so the filter was a no-op that just preserved the obsolete `format` field's presence.

### DeckOverlay--Scroll-UI adoption (enhanced-v3)
- `src/pages/scroll/pitch/enhanced-v3/index.astro` swapped its direct `<SlideRankPill>` mount for `<DeckOverlay--Scroll-UI deckSlug="pitch" variantSlug="enhanced-v3" />`. Zero behavior change; opens the door for future overlay additions (variant cycling, presenter notes, telemetry) via named slots without touching this file again.

### Consumer-side sitemap convention
- `context-v/sitemap/README.md` codifies the consumer-side half of the DidiDecks sitemap convention — what an artifact mini-spec looks like, the `injects_into_shell_slot:` frontmatter pattern that ties a per-slide component file to the shell route that renders it, and the wikilink discipline back to the universal shell artifacts at `../../../../ai-labs/dididecks-ai/context-v/sitemap/`.
- **5 seed mini-specs**: one route (`routes/scroll-pitch-enhanced-v3.md`) and three per-slide components (`slides/enhanced-v3/01-cover.md`, `15-ask.md`, `05-bottleneck.md` with the decompose-stub edge case flagged).
- Naming-fuzziness reminder in the README: Scroll-UI sections and Play-UI standalone files are **different files with different constraints**, not two views of the same content — coordinated by slot identity, requiring a recreate-don't-extract conversion.

### 2026-05-16 — DeckOverlay adoption, /play chooser rebuild, SLOTS["proto"]

The DeckOverlay components landed in the shell that morning. Adoption in chroma's enhanced-v3 scroll route was a one-line import swap (`f7b5cd9`). Same commit scaffolded the consumer-side context-v/sitemap/.

The `/play/` chooser had been broken since the shell shipped — its body said `(decks would render here)` as a placeholder, and its filter `decksByFormat("play")` always returned `[]` because pitch was registered as `format: "scroll"`. The fix (`5c4497b`) was bigger than the surface: the entire `DeckFormat` axis was a v0.1 legacy from before `@dididecks/shell` existed and didn't fit the corrected model where every deck has *both* a Scroll-UI implementation and an as-many-slots-as-have-files Play-UI implementation. Drop the type, drop the helper, drop the filter, port the card grid pattern from `/scroll/index.astro`. The first build attempt broke because `import.meta.glob` lived in `decks.ts` and the shell's registry-loader evaluates that file in a non-Vite context — split it into `play-availability.ts` and the build went green.

The chooser then linked the founder to `/play/pitch/proto/01/`, which 404'd. Diagnosis (`8dae60b`): proto had 13 per-slide files but no `SLOTS["proto"]` entry in `slides.ts`, so the shell's `getStaticPaths` (which reads SLOTS, not the file system) emitted nothing for proto. The fix was three coupled changes: (1) populate `SLOTS["proto"]` with 13 entries matching the actual file structure (slot 06 skipped); (2) AND-intersect file × SLOTS in `playableSlotsByVariant()` so the chooser is structurally incapable of linking anywhere the shell can't render; (3) flip the chooser's primary CTA to the newest playable variant rather than the first.

End state: all 46 slot pages emit. `/play/` works. The chooser lands you at enhanced-v3 by default with proto and enhanced-v2 as secondary chips. Typing any reachable URL works. The only remaining 404 is `/play/pitch/enhanced-v1/` (enhanced-v1 has no SLOTS, no per-slide files — that's A++.1 territory).

## How it works (or "Under the hood")

### Runtime-detect Play-UI availability via `import.meta.glob`

```ts
// src/data/play-availability.ts (excerpt)
import { SLOTS } from "./slides";

const SLIDE_FILES = import.meta.glob("/src/components/slides/**/*.astro");

export function playableSlotsByVariant(): Map<string, string[]> {
  // Step 1: Discover files (the consumer-side truth source)
  const filesByVariant = new Map<string, Set<string>>();
  for (const key of Object.keys(SLIDE_FILES)) {
    const m = key.match(/\/src\/components\/slides\/([^/]+)\/(\d+[a-z]?)-/);
    if (!m) continue;
    const [, variant, slot] = m;
    const set = filesByVariant.get(variant) ?? new Set<string>();
    set.add(slot);
    filesByVariant.set(variant, set);
  }

  // Step 2: AND-intersect with SLOTS (the shell-side truth source)
  const result = new Map<string, string[]>();
  for (const [variant, slots] of Object.entries(SLOTS)) {
    const fileSet = filesByVariant.get(variant);
    if (!fileSet) continue;
    const playable = slots
      .filter((s) => fileSet.has(s.slot))
      .map((s) => s.slot);
    if (playable.length > 0) result.set(variant, playable);
  }
  return result;
}
```

The intersection is load-bearing. Without it, the chooser could link to:
- Routes where files exist but SLOTS doesn't list them → shell's `getStaticPaths` skips → 404.
- Routes where SLOTS lists slots but files don't exist → shell renders DecomposeFirstPlaceholder → "playable" but not yet authored, surface-level misleading.

The intersection means: a deck shows up in the chooser only when there's at least one slot that the shell will render with real content.

### Why `play-availability.ts` lives outside `decks.ts`

The shell's integration evaluates the consumer's `decks.ts` to discover decks at build time. It uses an esbuild-backed data-URL evaluator (`registry-loader.ts` in the shell) — not Vite. In that pathway, `import.meta.glob` is undefined and throws `(intermediate value).glob is not a function` at build time. Splitting the glob into a sibling file the shell never imports keeps the registry-loader happy while letting Vite-evaluated routes use the glob normally.
