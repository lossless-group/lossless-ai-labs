## Why Care?

Every Lossless AI product we ship — investment-memo orchestration, slide-deck operating systems, fundraise materials, the augment-it experiment that's about to land — needs to know who's looking at it. The instinct is to reach for [Clerk](https://clerk.com), [Auth0](https://auth0.com), [WorkOS](https://workos.com), or one of the other auth-as-a-service vendors whose pricing pages assume you've raised a Series B and your users number in the hundreds of thousands. We are not them. We are a small team with maybe tens of named humans at a handful of client firms, and we want code we can read on a flight.

The [Shared-Auth-for-Applied-AI-Labs exploration](../../../../context-v/explorations/Shared-Auth-for-Applied-AI-Labs.md) framed the cost / control tradeoff in detail earlier this month. The conclusion: ~800 lines of owned auth code, libSQL-backed sessions, GitHub OAuth via the small [`arctic`](https://github.com/pilcrowOnPaper/arctic) library, an opaque `lossless_id` UUIDv7 that joins a person across apps, and a per-app `AuthEvent` outbox that a future ingester will roll up into one cross-app view. Three sessions, plus a package extraction, plus a calmstorm migration. Maybe a week of focused work.

Tonight we collapsed most of that into a single session — landing the full surface in chroma-decks (which was greenfield, no auth code, output set to `"static"`), with the four Session-2 additions baked into the initial schema rather than added as deltas. We didn't extract a package, and we deliberately didn't touch calmstorm beyond a small `app_slug` parity commit. The bet: chroma being greenfield makes it the cleaner reference for any future extraction; calmstorm's pre-existing schema can migrate later when the package surface is empirically validated.

If you are a chroma investor, when you visit a deck URL after this lands in prod, you'll either get bounced to a brand-themed gate (passcode flow, "Continue with GitHub" if you have a Lossless team identity) or — once you've authenticated once — straight into the deck with your session cookie carrying enough context that we know who you are when you scroll the next slide. If you're a Lossless teammate reading this, the same primitives ship into memopop-ai and augment-it next, with no schema changes required — the convention is locked.

## The arc: how we ended up here

The first instruction was *"make progress on the Shared-Auth exploration"* and a pointer at the doc. The exploration's three-session plan was clear:

1. Audit calmstorm's existing auth (Session 1, already done — see [Calmstorm-Auth-Inventory.md](../../../context-v/specs/Calmstorm-Auth-Inventory.md))
2. Add four missing pieces to calmstorm (Session 2 — OAuth, Organization/Membership, lossless_id, app_slug)
3. Stand up `lossless-auth-core` package + wire chroma-decks as the second consumer (Session 3)

I started Session 2.4 — the smallest of the four additions, `app_slug` on AuthEvent — in **calmstorm-decks**. About 15 lines across 5 files. Then the redirect:

> "wait I had wanted to implement this in chroma-decks"

Chroma at that point had **zero auth code**. `output: "static"`, no `@astrojs/db`, no `.env`, no sessions, no middleware. Reframing what "do this in chroma" meant: not "do Session 2.4 there too" but "install the entire auth surface from scratch, with the four Session-2 additions baked in from day one." Greenfield made that cleaner — no migration of an existing TEXT `org` column, no live sessions to preserve.

I scoped what that actually entailed (SSR switch, db setup, schema, helpers, middleware, /access UI, OAuth) and wrote it up as a [6-phase plan](../context-v/plans/Install-Auth-Surface-from-Calmstorm-Pattern.md) in chroma-decks' own `context-v/plans/`. Then started executing.

### The Phase 1 trap (and the calmstorm fix that saved us)

Phase 1 wired SSR and added `export const prerender = true` to all 8 existing deck pages on the theory that we'd get CDN-fast loads while the middleware would still gate access. Sold the user on "SSG-fast + SSR-gated, no compromise."

Wrong.

Calmstorm landed a commit earlier today (`677417f5`) fixing **exactly** this: under `output: "server"` on Vercel, prerendered pages are CDN-served as static HTML and the middleware function never runs for them in production. Anyone with a URL bypasses the gate. The commit body said it plainly:

> `/play/variant/[variant].astro` and `/play/section/[slot].astro` shipped with `export const prerender = true`. With `output: "server"`, that means the Vercel CDN serves their static HTML directly — the access middleware never runs in production, so anyone with the URL plays the deck.

I caught this between Phase 1 and Phase 4 by reading calmstorm's recent git log before writing the middleware. Stripped `prerender = true` from all 8 chroma pages, folded the correction into the Phase 4 commit (`ba57630`), updated the plan file to reflect the corrected stance, and moved on. The tradeoff is ~50ms of per-request SSR cost on warm Vercel functions instead of CDN-direct serving. For fundraise-deck traffic — low concurrency, returning visitors with warm sessions — irrelevant.

### Phase 5: the gate becomes functional, and the passcode-dash gotcha

Phase 5 ported the `/access` UI and the `/api/access/verify` + `/api/access/logout` routes from calmstorm, with chroma branding (lowercase wordmark, IBM Plex Mono eyebrow, orange accent on focus + errors per [DESIGN.md](../DESIGN.md)) and the lossless_id + Membership-to-Chroma-org logic wired in.

Quick gotcha when testing: the passcode helper splits the submitted string on the **first dash** to extract an optional `shared_label` suffix. So if your `.env` says `ADMIN_PASSCODE=chroma-test-admin`, the helper parses that as base=`chroma`, label=`test-admin`, and nothing ever matches. The fix is to pick passcodes with no dashes (`chromaadmin`, `chromaviewer`); users can still append a label at submission time (`chromaviewer-airstreet`) to attribute the session.

The deck-shell built off without errors, the middleware bounced unauthenticated `/` requests to `/access`, the form posted to `/api/access/verify`, the passcode matched (after the dash debug), a `cd_session` cookie was set, and a redirect back to `/` landed the user in the deck. End-to-end functional.

### Phase 6: arctic earns its keep

The exploration recommended [`arctic`](https://github.com/pilcrowOnPaper/arctic) — a small, scoped OAuth-handshake library by the Lucia author. The user pushed back, citing the exploration's own "code we own" stance: *"Why do we need to use arctic? I thought we were going to roll our own."*

Honest assessment: for **GitHub-only**, hand-rolling is ~80 LOC of `fetch()` boilerplate, zero dependency, fully inspectable. `arctic` earns its keep when you add Google + Apple + Microsoft (each with their own PKCE / ID-token / domain-hint / signed-JWT-secret quirks). For now we said GitHub-only, Google Workspace deferred.

User picked `arctic` anyway after weighing the tradeoff. Fine. ~40 LOC of route code consumes arctic's `GitHub` class for authorize-URL construction + token exchange; everything else (state-cookie CSRF defense, `/user/emails` fallback for users with private emails, `OAuthAccount` upsert, `resolveOrCreateIdentity` integration, Membership attachment, session minting) stays in our own code.

The callback route is the substantial one — 270 LOC — and it's where the `lossless_id` contract from the parent exploration earns its keep: a returning user who first signed in via magic-link gets the same `lossless_id` when they later sign in via GitHub, because we join on email. First-time users get a fresh UUIDv7 minted by `resolveOrCreateIdentity()`. The function name is the source of truth: resolve-or-create, idempotent, one Identity per primary_email per app.

### The data-assets prerender leak

While testing the gate end-to-end, the dev server's terminal showed something concerning during builds and dev navigations:

```
prerendering static routes
  ├─ /data-assets/companies/index.html
  ├─ /data-assets/people/index.html
[middleware] /data-assets/people/: no cookie → /access
```

Two clues: (1) those routes were being **prerendered** (so they'd be CDN-served in prod and skip middleware), and (2) the middleware ran during build's prerender step and tried to bounce them — which would have baked a redirect into the static HTML, possibly. Either way: the routes were ungated in prod.

`/data-assets/companies` renders every portfolio company across all four VC firms chroma is fundraising from — Air Street Capital, AIX Ventures, Bloomberg Beta, Quiet Capital — grouped by firm. Mostly already-public info but tabular and machine-readable.

`/data-assets/people` renders chroma's team + advisor graph — Naval Ravikant, Nat Friedman, Spencer Kimball, Jeff Huber, Guillermo Rauch, on down — with photos, LinkedIn, role, firm affiliation, bio. **That is sensitive.** Knowing who's helping chroma is competitive intel and signal.

Routes live in `@dididecks/shell`, not in chroma. Two-line patch on each: `export const prerender = false` (commit `7cb5cc7` on dididecks-ai). Now the consumer's middleware actually runs on those routes in prod. Calmstorm benefits at no extra cost.

### Organization naming: domain-as-id

Late in the session: *"the organizations needed to generate at first, and we should put this in CLAUDE.md or something in dididecks, is `lossless-group` and `{client-name}` or we should just use the url/email domain so `lossless.group` and `trychroma.com`"*

The latter wins, hands down:

- **OAuth-natural.** A Google Workspace OAuth callback gives you `user@trychroma.com`. The org id is `email.split('@')[1]`. Zero translation table.
- **Globally unique.** Two firms can both want the slug `"studio"`; nobody else gets your domain.
- **Self-documenting.** `organization_id = "trychroma.com"` requires no mental mapping.
- **Cross-app rollup keys naturally.** When memopop and chroma and (eventually) augment-it all stamp memberships against `trychroma.com`, the future rollup ingester joins on the same id across apps without any coordination.

`Organization.id` is the canonical domain. `Organization.slug` is the human handle (`chroma`, `lossless-group`, `calm-storm-ventures`). Every app seeds two rows on first boot: `lossless.group` + `{client-domain}`. Personal-email signups bucket as `id = "personal"`. Multi-domain firms get an `aliases` column when first encountered; not v1.

Locked into the parent exploration (`eb51759` on ai-labs main), pointer added to `dididecks-ai/CLAUDE.md`, applied to chroma-decks (`84481e7`). The previous `id = "chroma"` row was migrated by nuking the local DB and clearing the remote — acceptable because no real session data had accumulated yet.

### Sticky table headers, while we're in there

While the data-assets/people audit was open in a browser, the user noticed: *"the table header is not clear. I've found its generally good to have a sticky header, so when the user scrolls down they keep the header context to understand what they are looking at."*

The `<th>` already had `position: sticky; top: 53px;` — the author had intended it. But `.table-wrap { overflow-x: auto }` was establishing a scrolling-ancestor that trapped the sticky vertically (sticky finds its nearest scrolling ancestor, and `overflow-x:auto` makes the wrap that ancestor even with no vertical scroll). One-line fix: drop the overflow on the wrap; let the table push the page horizontally on narrow viewports. For audit pages that's fine. Commit `f4d12c7`.

## What this enables next

- **chroma can ship to investors with a real gate.** The passcode flow is the simplest credential path; GitHub OAuth unlocks when we register a GitHub OAuth App and drop the client id/secret into Vercel env.
- **calmstorm and chroma will join on `lossless_id` when the rollup ingester gets built.** The seam is there; the ingester is a weekend project when we have a real cross-app question to answer (likely: "who's looked at both decks").
- **memopop-ai and augment-it install this same surface** with the convention now locked. Each app keeps its own DB; each stamps its own `app_slug`; each seeds `lossless.group` + its own `{client-domain}` org. No central service required for any single app to function.
- **The eventual `lossless-auth-core` package extraction** has a second consumer (chroma) ready to use it. Calmstorm becomes the third consumer, with one small migration (`cs_session` → standardized cookie name, role enum `admin` → `org_admin`). Until that extraction happens, both sites stay parallel — the same pattern, twice, validating the shape before we lock it.
- **The `@lossless.group` → `superuser` fast-path** is named as a follow-up: when an OAuth callback yields an email matching `lossless.group`, auto-attach `superuser` Membership to the `lossless.group` org. Small change in the OAuth callback; saves manual admin work for the operating team.
